# Решение упражнения 6 (трейс)

## Сценарий и инструменты

Задача агента - найти и исправить опечатку в README. Нужны два инструмента из
прошлых заданий: `read_file` (корзина allow) и `edit_file` (корзина ask). Опираюсь
на разделение из конспекта: модель это мозг, который только генерирует намерение,
а рантайм это руки и служба безопасности, который валидирует, проверяет политику,
исполняет и возвращает результат.

Важный нюанс API из примеров урока: блок `tool_result` кладётся в сообщение с
ролью `user`, но наполняет его НЕ человек, а рантайм. Поэтому роль `user` в трейсе
встречается в двух разных смыслах - реальный пользователь и обёртка для ответа
инструмента.

Ещё одна оговорка про формат. В реальном Messages API системный промпт это не
элемент массива `messages` с ролью `system`, а отдельный top-level параметр
запроса. Ниже я показываю его строкой трейса для наглядности полного контекста, но
в боевом вызове он лежит вне массива сообщений. Так же стоит держать в голове и
подтверждение на политику ask - рантайм спрашивает пользователя вне массива
`messages`, поэтому в коде-трейсе этот шаг помечен комментарием, а не отдельным
сообщением.

## Полный трейс сообщений

| № | Роль (API) | Кто работает     | Содержание                                                                                                    | Источник данных                                        |
|---|------------|------------------|---------------------------------------------------------------------------------------------------------------|--------------------------------------------------------|
| 0 | system     | рантайм собирает | системный промпт, роль агента, определения и схемы инструментов, политики allow/ask/block                     | конфиг агента и определения инструментов (разработчик) |
| 1 | user       | пользователь     | «найди и исправь опечатку в README»                                                                           | ввод пользователя                                      |
| 2 | assistant  | модель           | `tool_use` read_file{path: "README.md"}, `stop_reason=tool_use`                                               | генерация модели                                       |
| 3 | (рантайм)  | рантайм          | валидация аргументов, политика read_file=allow, путь не секрет, чтение файла                                  | проверка политики - внутренняя                         |
| 4 | user       | рантайм          | `tool_result` с текстом README                                                                                | файловая система (чтение README.md)                    |
| 5 | assistant  | модель           | текст «нашёл опечатку: устнавливать -> устанавливать» плюс `tool_use` edit_file{path, old_string, new_string} | генерация модели (опечатка найдена в данных из шага 4) |
| 6 | (рантайм)  | рантайм          | политика edit_file=ask, запрос подтверждения у пользователя                                                   | проверка политики - внутренняя                         |
| 7 | user       | пользователь     | подтверждение правки (ответ на ask)                                                                           | ввод пользователя                                      |
| 8 | user       | рантайм          | `tool_result` со статусом записи и diff                                                                       | файловая система (запись в README.md)                  |
| 9 | assistant  | модель           | финальный текст «опечатка исправлена», `stop_reason=end_turn`                                                 | генерация модели                                       |

Шаги 3 и 6 это чистая работа рантайма внутри цикла, они не отдельные сообщения в
истории, а действия между сообщениями - валидация, политика, исполнение. Я вынес
их строками, чтобы было видно, где рантайм принимает решения.

## Где работает модель, где рантайм

- Модель: шаги 2, 5, 9. Только генерирует намерения (`tool_use`) и финальный текст.
  Она не читает и не пишет файл сама, а просит рантайм это сделать.
- Рантайм: шаги 0, 3, 4, 6, 8. Сборка системного промпта, валидация, политика,
  фактическое чтение и запись, упаковка результата в `tool_result`.
- Пользователь: шаги 1 и 7. Постановка задачи и подтверждение мутации на ask.

## Источники данных

1. Конфиг агента и определения инструментов - системный промпт и схемы (шаг 0),
   готовит разработчик или рантайм.
2. Ввод пользователя - постановка задачи (шаг 1) и подтверждение правки (шаг 7).
3. Файловая система - содержимое README при чтении (шаг 4) и результат записи с
   diff (шаг 8). Это единственный внешний по отношению к диалогу источник фактов.
4. Генерация модели - шаги 2, 5, 9. Не внешний источник, а вывод модели на основе
   контекста, поэтому фактам отсюда верим только если они опираются на данные из
   `tool_result`.

Ключевой вывод: единственный достоверный внешний источник правды о файле - это
`tool_result` от рантайма (файловая система). Текст модели про опечатку валиден
лишь потому, что опирается на прочитанное на шаге 4, а не на догадку.

## Код: трейс и разметка источников

Трейс оформлен списком сообщений в формате Messages API, а чистая функция
размечает каждое сообщение по исполнителю и источнику.

```python
"""Трейс сценария исправления опечатки в README с разметкой источников

Список сообщений в формате Messages API плюс чистая функция annotate,
которая помечает каждое сообщение - кто работает и откуда данные
"""

from typing import Any

# полный трейс диалога, tool_result лежит в роли user но его пишет рантайм
# роль system тут показана строкой для наглядности, в реальном API это
# top-level параметр запроса а не элемент массива messages
TRACE: list[dict[str, Any]] = [
    {
        "role": "system",
        "content": "Ты агент-редактор. Инструменты: read_file, edit_file",
    },
    {
        "role": "user",
        "content": "Найди и исправь опечатку в README",
    },
    {
        "role": "assistant",
        "content": [
            {
                "type": "tool_use",
                "id": "t1",
                "name": "read_file",
                "input": {"path": "README.md"},
            }
        ],
    },
    {
        "role": "user",
        "content": [
            {
                "type": "tool_result",
                "tool_use_id": "t1",
                "content": "Как устнавливать пакет: ...",
            }
        ],
    },
    {
        "role": "assistant",
        "content": [
            {
                "type": "text",
                "text": "Нашел опечатку устнавливать -> устанавливать",
            },
            {
                "type": "tool_use",
                "id": "t2",
                "name": "edit_file",
                "input": {
                    "path": "README.md",
                    "old_string": "устнавливать",
                    "new_string": "устанавливать",
                },
            },
        ],
    },
    # перед этим tool_result рантайм по политике ask запросил у пользователя
    # подтверждение правки, оно идет вне массива messages и тут не показано
    {
        "role": "user",
        "content": [
            {
                "type": "tool_result",
                "tool_use_id": "t2",
                "content": "{\"status\": \"ok\", \"replaced\": 1}",
            }
        ],
    },
    {
        "role": "assistant",
        "content": [
            {"type": "text", "text": "Опечатка исправлена"}
        ],
    },
]


def _has_tool_result(content: Any) -> bool:
    """Проверяет, есть ли в содержимом блок tool_result"""
    if not isinstance(content, list):
        return False
    return any(
        isinstance(b, dict) and b.get("type") == "tool_result"
            for b in content
    )


def annotate(message: dict[str, Any]) -> dict[str, str]:
    """Размечает сообщение по исполнителю и источнику данных

    Возвращает словарь с полями actor и source. Сообщение на вход не
    меняется. tool_result в роли user относим к рантайму и файловой системе
    """
    role = message.get("role")
    content = message.get("content")

    if role == "system":
        return {"actor": "рантайм", "source": "конфиг агента"}
    if role == "assistant":
        return {"actor": "модель", "source": "генерация модели"}
    if role == "user":
        if _has_tool_result(content):
            return {"actor": "рантайм", "source": "файловая система"}
        return {"actor": "пользователь", "source": "ввод пользователя"}
    return {"actor": "неизвестно", "source": "неизвестно"}
```

Что показывает код:

1. `TRACE` это тот же сценарий в формате Messages API - видно чередование
   `assistant` с `tool_use` и `user` с `tool_result`.
2. `annotate` чистая и по роли и наличию `tool_result` разводит три исполнителя:
   модель, рантайм и пользователя, не путая рантайм-обёртку с реальным человеком.
3. Это машинная версия таблицы выше - тот же разбор можно прогнать по реальному
   логу постфактум.
