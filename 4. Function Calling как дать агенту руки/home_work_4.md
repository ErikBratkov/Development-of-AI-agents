# Решение упражнения 4 (политика)

## Базовая логика классификации

Опираюсь на конспект. Read-only обычно безопасно - allow, но не для секретов и
недоверенных путей из-за prompt injection. Мутации в production должны быть ask или
block, allow для них допустима только после долгого тестирования. Проект сейчас на
стадии подготовки к MVP и до production не дошёл, поэтому к мутациям отношусь
строго. У каждой корзины указываю базовое значение и условие эскалации или
смягчения - политику считает Runtime, а не модель.

## Таблица классификации

| Инструмент    | Тип операции            | Корзина | Обоснование и условие перехода                                                                                    |
|---------------|-------------------------|---------|-------------------------------------------------------------------------------------------------------------------|
| `read_file`   | чтение                  | allow   | чтение безопасно, но эскалация в block на `.env`, ключи, секреты и недоверенные пути - защита от prompt injection |
| `edit_file`   | мутация, локальная      | ask     | меняет файлы, до production спрашиваем пользователя, откатываемо через git                                        |
| `run_test`    | исполнение кода         | ask     | запускает произвольный код с побочными эффектами, смягчение до allow только в изолированной песочнице             |
| `delete_file` | мутация, разрушительная | block   | необратимое удаление, смягчение до ask если есть корзина или мягкое удаление с восстановлением                    |
| `send_email`  | мутация, внешняя        | ask     | необратимая отправка наружу, эскалация в block при массовой рассылке или внешних адресах                          |
| `git_push`    | мутация, внешняя, общая | ask     | меняет общий репозиторий, эскалация в block для защищённых веток вроде main                                       |

Итог по корзинам - allow: `read_file`. ask: `edit_file`, `run_test`, `send_email`,
`git_push`. block: `delete_file`. Остальные переходы между корзинами зависят от
контекста вызова и считаются в Runtime.

## Два пограничных случая

### 1. `run_test` - граница allow и ask

Соблазн положить в allow, ведь «прогнать тесты» звучит безобидно и часто это
read-only по смыслу. Но `run_test` исполняет произвольный код: тест может ходить в
сеть, писать в БД, трогать файлы, жечь деньги на внешних API. Это уже не чтение, а
исполнение с непредсказуемыми побочными эффектами, поэтому базово ask. Смягчаю до
allow только когда прогон идёт в изолированной песочнице без доступа наружу и к
проду - тогда побочные эффекты ограничены и риск приемлемый.

### 2. `delete_file` - граница ask и block

Это мутация, как `edit_file`, и напрашивается ask. Разница в обратимости.
Правку файла видно в git и легко откатить, а удаление часто необратимо, особенно
если модель ошиблась с путём из задания 2. Цена ошибки несимметрична, поэтому
базово block. Смягчаю до ask, когда удаление мягкое и восстановимое - перенос в
корзину или soft-delete с возможностью отката. Тогда необратимости нет и хватает
подтверждения пользователя.

## Код политики

Политику оформляю как чистую функцию Runtime, которая по инструменту и контексту
возвращает корзину. Это и есть guardrail из конспекта - решение принимает код, а
не модель.

```python
"""Политика доступа к инструментам в стиле guardrails из урока 4

Чистая функция classify по имени инструмента и контексту вызова возвращает
корзину allow, ask или block. Контекст не модифицируется
"""

from typing import Any

# базовые корзины для каждого инструмента
BASE_POLICY = {
    "read_file": "allow",
    "edit_file": "ask",
    "run_test": "ask",
    "delete_file": "block",
    "send_email": "ask",
    "git_push": "ask",
}

# части пути, чтение которых запрещаем из за риска утечки секретов
SECRET_HINTS = (".env", "secret", "credential", "id_rsa", ".key")

# ветки, push в которые требует жесткой блокировки
PROTECTED_BRANCHES = ("main", "master", "release")

# больше этого числа адресатов считаем рассылку массовой и блокируем
MAX_DIRECT_RECIPIENTS = 5


def _looks_secret(path: str) -> bool:
    """Грубо проверяет, похож ли путь на файл с секретами"""
    low = path.lower()
    return any(hint in low for hint in SECRET_HINTS)


def classify(tool: str, context: dict[str, Any] | None = None) -> str:
    """Возвращает корзину allow, ask или block для вызова инструмента

    Учитывает контекст - путь файла, песочницу, восстановимость удаления,
    адресатов письма и ветку push. Неизвестный инструмент всегда block,
    контекст на вход не меняется
    """
    ctx = context or {}
    base = BASE_POLICY.get(tool)
    if base is None:
        # незнакомый инструмент безопаснее запретить чем пустить
        return "block"

    if tool == "read_file":
        # чтение секретов поднимаем до block даже при базовом allow
        if _looks_secret(str(ctx.get("path", ""))):
            return "block"
        return "allow"

    if tool == "run_test":
        # изолированная песочница ограничивает побочные эффекты
        if ctx.get("sandbox"):
            return "allow"
        return "ask"

    if tool == "delete_file":
        # мягкое восстановимое удаление снимаем с block до ask
        if ctx.get("recoverable"):
            return "ask"
        return "block"

    if tool == "send_email":
        # массовая или внешняя рассылка опаснее точечной
        if ctx.get("recipients", 0) > MAX_DIRECT_RECIPIENTS \
                or ctx.get("external"):
            return "block"
        return "ask"

    if tool == "git_push":
        # push в защищенную ветку блокируем
        if ctx.get("branch") in PROTECTED_BRANCHES:
            return "block"
        return "ask"

    return base
```

Что показывает код:

1. `BASE_POLICY` отражает строки таблицы, а `classify` накладывает контекстные
   переходы - эскалацию (`read_file` на секретах, `git_push` в main) и смягчение
   (`run_test` в песочнице, `delete_file` с восстановлением).
2. Незнакомый инструмент уходит в block - по умолчанию запрещаем то, чего не
   знаем, а не пускаем.
3. Функция чистая, контекст на вход не меняется, что удобно для тестов и аудита
   решений политики.
