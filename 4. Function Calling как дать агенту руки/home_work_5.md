# Домашнее задание к уроку №4 "Function Calling: как дать агенту руки"

Упражнение 5 (идемпотентность)

## Цель:

Придумайте ключ идемпотентности для `create_ticket`, который защитит от дублей при анализе одного и того же лога дважды,
но не помешает создать два разных тикета по двум разным проблемам.

# Решение упражнения 5 (идемпотентность)

## Чего хотим от ключа

Конспект называет идемпотентность критичной для мутаций вроде создания тикетов и
защищает от дублей, когда модель или сеть зависли и отправили запрос дважды.
Задача ставит два условия одновременно:

- анализ одного и того же лога дважды НЕ должен плодить дубль одного тикета
- две разные проблемы должны давать два разных тикета

Отсюда вывод о природе ключа - он обязан выводиться из СМЫСЛА проблемы, а не из
момента или случайности вызова.

## Что НЕ подходит

- UUID или timestamp на каждый вызов. Повторный анализ того же лога даст другой
  ключ, и появится дубль. Прямо противоречит первому условию.
- Один лишь идентификатор лога (`log_id`). Тогда у двух разных проблем из одного
  лога ключ совпадёт, и второй тикет не создастся. Противоречит второму условию.

## Предлагаемый ключ

Ключ - это хеш от нормализованной сигнатуры проблемы:

```
idempotency_key = sha256(project | error_type | location | test | norm_message)
```

Составляющие подобраны так, чтобы одна и та же проблема всегда давала один ключ, а
разные проблемы - разные:

- `project` - ключ проекта, разводит одинаковые ошибки в разных проектах
- `error_type` - класс ошибки, например AssertionError или TimeoutError
- `location` - файл и при наличии функция, где упало
- `test` - имя упавшего теста или имя операции
- `norm_message` - текст ошибки после нормализации

Главная деталь - нормализация сообщения. Сырой текст от LLM при повторном анализе
может быть перефразирован, и хеш разъедется, дав ложный дубль-тикет. Поэтому из
сообщения вычищаем волатильные части - числа, hex-адреса, uuid, временные метки,
временные пути - и приводим к нижнему регистру. Остаётся стабильный скелет
проблемы.

Так выполняются оба условия. Тот же лог дважды - те же сигнатуры, те же ключи,
дублей нет. Две разные проблемы - разный `error_type`, `location` или
`norm_message`, значит разные ключи и два тикета.

## Что делает Runtime с ключом

`create_ticket` принимает `idempotency_key`. Перед созданием Runtime смотрит в
хранилище ключей. Если ключ уже есть - возвращает id ранее созданного тикета и
ничего не создаёт (операция идемпотентна). Если ключа нет - создаёт тикет,
сохраняет пару ключ-id и возвращает новый id. Это закрывает и сценарий двойной
отправки из-за зависшей сети.

Про границы. Сам лог намеренно НЕ кладу в ключ - иначе одна и та же повторяющаяся
проблема из разных прогонов плодила бы тикеты. Если же по политике повтор в новом
логе должен заводить свежий тикет, в ключ добавляется отпечаток лога, а старым
ключам имеет смысл задать TTL, чтобы окно дедупликации не было вечным.

## Код

Ключ и дедупликация показаны в стиле Runtime из урока - чистые функции плюс
хранилище.

```python
"""Идемпотентность для create_ticket в стиле guardrails урока 4

Ключ выводится из нормализованной сигнатуры проблемы, поэтому повторный
анализ того же лога не плодит дубль, а разные проблемы дают разные ключи
"""

import hashlib
import re
from typing import Any

# волатильные куски сообщения, которые мешают стабильности ключа
_HEX = re.compile(r"0x[0-9a-f]+", re.I)
_UUID = re.compile(
    r"[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-"
    r"[0-9a-f]{4}-[0-9a-f]{12}",
    re.I,
)
# временная метка вида 2026-06-15t10:30:00 или с пробелом вместо t
_TIME = re.compile(r"\d{4}-\d{2}-\d{2}[t ]\d{2}:\d{2}:\d{2}")
# временные пути запусков, например /tmp/pytest-of-user/run123
_TMP = re.compile(r"/tmp/\S+")
_NUM = re.compile(r"\d+")
_SPACE = re.compile(r"\s+")


def normalize_message(message: str) -> str:
    """Чистит текст ошибки от волатильных частей для стабильного ключа

    Убирает uuid, hex-адреса, временные метки, tmp-пути и числа, схлопывает
    пробелы и опускает регистр. Порядок важен - uuid, время и пути снимаем до
    общего удаления цифр. Возвращает новую строку, аргумент не меняется
    """
    text = message.lower()
    text = _UUID.sub("#", text)
    text = _HEX.sub("#", text)
    text = _TIME.sub("#", text)
    text = _TMP.sub("#", text)
    text = _NUM.sub("#", text)
    return _SPACE.sub(" ", text).strip()


def make_idempotency_key(problem: dict[str, Any]) -> str:
    """Строит ключ идемпотентности из сигнатуры проблемы

    Берет проект, класс ошибки, место, тест и нормализованное сообщение.
    Одна и та же проблема дает один ключ, разные - разные. Вход не меняется
    """
    parts = [
        str(problem.get("project", "")),
        str(problem.get("error_type", "")),
        str(problem.get("location", "")),
        str(problem.get("test", "")),
        normalize_message(str(problem.get("message", ""))),
    ]
    raw = "|".join(parts)
    return hashlib.sha256(raw.encode("utf-8")).hexdigest()


class TicketStore:
    """Хранилище тикетов с дедупликацией по ключу идемпотентности"""

    def __init__(self) -> None:
        """Создает пустое хранилище ключей и счетчик тикетов"""
        self._by_key: dict[str, str] = {}
        self._counter = 0

    def create_ticket(self, problem: dict[str, Any]) -> dict[str, Any]:
        """Создает тикет или возвращает существующий по тому же ключу

        Если ключ уже виден - отдает прежний id и ставит deduped, иначе
        заводит новый тикет. Защищает от дублей и двойной отправки
        """
        key = make_idempotency_key(problem)
        existing = self._by_key.get(key)
        if existing is not None:
            return {"ticket_id": existing, "deduped": True}

        self._counter += 1
        ticket_id = f"RK-{self._counter}"
        self._by_key[key] = ticket_id
        return {"ticket_id": ticket_id, "deduped": False}
```

Что показывает код:

1. `normalize_message` гасит перефразировку и волатильные числа, поэтому повторный
   анализ того же лога даёт тот же ключ.
2. `make_idempotency_key` собирает ключ из стабильных дискриминаторов проблемы -
   разные проблемы разъезжаются по разным ключам.
3. `TicketStore.create_ticket` на известном ключе возвращает прежний id с флагом
   `deduped`, закрывая дубли и двойную отправку, а обе функции ключа чистые.
