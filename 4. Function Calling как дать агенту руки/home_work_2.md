# Домашнее задание к уроку №4 "Function Calling: как дать агенту руки"

Упражнение 2 (диагностика)

## Цель:

Агент периодически читает не тот файл и иногда выполняет лишний шаг чтения.
Опишите, к какому(им) шагу(ам) цикла относится проблема и как бы вы её локализовали по трейсу.

# Решение упражнения 2 (диагностика)

## Шаги агентного цикла

За основу беру цикл из примера 2 урока. Одна итерация состоит из шагов:

1. Модель видит контекст (system + история) и генерирует намерение - блок
   `tool_use` с именем инструмента и аргументами. Это решение модели, а не
   исполнение.
2. Runtime разбирает `tool_use`, валидирует аргументы по схеме, проверяет права.
3. Runtime исполняет инструмент - в нашем случае резолвит путь и читает файл.
4. Runtime формирует `tool_result` (при необходимости обрезает) и дописывает его в
   `messages` напротив нужного `tool_use_id`.
5. Модель получает `tool_result` и решает - продолжать (новый `tool_use`) или
   завершить (`end_turn`).
6. Повтор шагов 1-5 пока `stop_reason` не станет `end_turn`.

Два симптома из задания относятся к разным шагам, поэтому разбираю их отдельно.

## Симптом А. Читает не тот файл

Это сбой выбора цели чтения. Виновником может быть шаг 1 или шаги 2-3, и это
принципиально разные баги.

- Шаг 1 (сторона модели). Модель сама положила в `tool_use.input` неверный путь.
  Типовые причины - размытое описание инструмента, неоднозначная ссылка на файл в
  запросе, устаревший путь, скопированный из ранней части истории, или просто
  галлюцинация имени. Лечится схемой и описанием, а не кодом Runtime.
- Шаги 2-3 (сторона Runtime). Модель попросила корректный путь, но Runtime открыл
  другой - неверный `cwd`, кривая нормализация относительного пути, рассинхрон
  symlink, гонка кеша. Тут модель не виновата.

Ключ к разделению - сравнить, что модель ЗАПРОСИЛА (`tool_use.input.path`), с тем,
что Runtime реально ОТКРЫЛ (абсолютный путь из лога исполнения). Совпали - баг на
шаге 1, разошлись - баг на шагах 2-3.

## Симптом Б. Лишний шаг чтения

Это сбой условия продолжения цикла - шаг 5, и часто завязан на шаг 4 (как
собирается история). Дубль чтения одного и того же файла обычно означает одно из:

1. Параллельные вызовы не отключены, и модель в одном ответе выдала два одинаковых
   `tool_use`. Это видно в рамках одной итерации (лечится
   `disable_parallel_tool_use`).
2. Результат первого чтения не дошёл до модели - `tool_result` не дописан в
   `messages`, прилетел с `is_error`, или обрезан до пустоты на шаге 4. Модель не
   видит данных и читает снова.
3. Результат дошёл, но описание инструмента провоцирует перепроверку, либо контент
   урезан так, что модель считает его неполным и идёт за остатком.

Чтобы понять, шаг 4 это или шаг 5 - проверяю, был ли валидный `tool_result` у
первого чтения к моменту второго. Есть валидный результат, а модель всё равно
перечитала - проблема решения модели (шаг 5). Результата нет или он битый -
проблема сборки истории (шаг 4).

## Как локализую по трейсу

Минимальный трейс должен логировать по каждой итерации:

- номер итерации и `stop_reason`
- все блоки `tool_use`: `id`, `name`, `input` (полностью)
- все `tool_result`: `tool_use_id`, признак `is_error`, длину контента, флаг
  обрезки
- сторону Runtime для чтения: запрошенный путь, итоговый абсолютный путь, сколько
  байт прочитано, `cwd`

Алгоритм разбора:

1. Симптом А. Для каждого чтения сверяю `input.path` с абсолютным путём, который
   открыл Runtime. Расхождение - локализовано на шагах 2-3. Совпадение - смотрю
   контекст перед вызовом: упоминался ли правильный путь в запросе или в прошлом
   `tool_result`, не утащила ли модель старый путь из истории. Это локализует баг
   на шаге 1.
2. Симптом Б. Группирую чтения по паре (`name`, нормализованный `input`). Если
   повтор внутри одной итерации - параллельные вызовы, шаг 1. Если повтор между
   итерациями - проверяю у первого чтения наличие парного валидного `tool_result`
   (по `tool_use_id`). Нет пары или `is_error`/пустой - шаг 4. Пара есть и валидна
   - шаг 5.
3. Дополнительно ищу осиротевшие `tool_use` без `tool_result` - это прямой признак
   поломки сборки истории на шаге 4.

## Небольшой помощник для разбора трейса

Утилита прогоняет трейс и помечает оба симптома, чтобы не искать глазами.

```python
"""Помощник локализации проблем чтения по трейсу агентного цикла

Трейс - список итераций. Каждая итерация это словарь с блоками tool_use,
которые выдала модель, и tool_result, которые вернул Runtime. Утилита не
меняет вход и возвращает список находок с привязкой к шагу цикла
"""

import json
from typing import Any


def _read_key(block: dict[str, Any]) -> str:
    """Строит ключ чтения из имени инструмента и аргументов

    Аргументы сериализуем с сортировкой ключей, чтобы одинаковые по сути
    вызовы давали одинаковый ключ независимо от порядка полей
    """
    args = json.dumps(block.get("input", {}), sort_keys=True,
                      ensure_ascii=False)
    return block["name"] + " " + args


def find_issues(
    trace: list[dict[str, Any]],
    read_tool: str = "read_file",
) -> list[dict[str, Any]]:
    """Находит чтение не того файла и лишние чтения, привязывая к шагам

    Возвращает список находок вида {symptom, step, detail}. Симптом A -
    расхождение запрошенного и открытого пути. Симптом B - повтор того же
    чтения. Вход не модифицируется
    """
    findings: list[dict[str, Any]] = []
    # ключи чтений из ПРЕДЫДУЩИХ итераций с их итогом
    seen: dict[str, bool] = {}

    for step in trace:
        uses = [b for b in step.get("tool_use", [])
                if b.get("name") == read_tool]
        results = {r["tool_use_id"]: r
                   for r in step.get("tool_result", [])}

        # симптом Б внутри одной итерации - параллельные одинаковые вызовы
        # помечаем по одному разу на повторяющийся ключ
        keys_here = [_read_key(b) for b in uses]
        dupes = sorted({k for k in keys_here if keys_here.count(k) > 1})
        for key in dupes:
            findings.append({
                "symptom": "B",
                "step": "шаг 1 (параллельные вызовы)",
                "detail": key,
            })

        for block in uses:
            # симптом А - сверяем запрошенный и реально открытый путь
            opened = block.get("opened_path")
            wanted = block.get("input", {}).get("path")
            if opened is not None and wanted is not None \
                    and opened != wanted:
                findings.append({
                    "symptom": "A",
                    "step": "шаги 2-3 (резолв пути в Runtime)",
                    "detail": f"просил {wanted}, открыл {opened}",
                })

            # симптом Б между итерациями - повтор ключа из прошлых итераций
            key = _read_key(block)
            if key in seen:
                if seen[key]:
                    step_label = "шаг 5 (решение модели)"
                else:
                    step_label = "шаг 4 (сборка истории)"
                findings.append({
                    "symptom": "B",
                    "step": step_label,
                    "detail": key,
                })

        # итоги текущей итерации фиксируем после ее полного разбора чтобы
        # дубли внутри одного ответа не считались повтором между итерациями
        for block in uses:
            res = results.get(block.get("id"))
            ok = bool(res) and not res.get("is_error") \
                and res.get("content_len", 0) > 0
            seen[_read_key(block)] = ok

    return findings
```

Что показывает помощник:

1. Расхождение `input.path` и `opened_path` сразу относит симптом А к Runtime
   (шаги 2-3), а совпадение оставляет его на шаге 1 для ручной проверки контекста.
2. Повтор ключа чтения внутри одной итерации - это параллельные вызовы (шаг 1),
   а между итерациями утилита смотрит на итог прошлого чтения: был валидный
   результат - значит виновато решение модели (шаг 5), не было - значит поломка
   сборки истории (шаг 4).
3. Функция чистая, трейс на вход не меняется, что удобно гонять на сохранённых
   логах постфактум.

## Краткие выводы и фиксы

- Симптом А, шаг 1 - усилить описание инструмента и схему пути, передавать модели
  точные пути в `tool_result`, чистить устаревшие пути из контекста.
- Симптом А, шаги 2-3 - чинить резолв пути и `cwd` в Runtime, логировать
  абсолютный открытый путь.
- Симптом Б, шаг 1 - выставить `disable_parallel_tool_use` если дубли в одном
  ответе не нужны.
- Симптом Б, шаг 4 - гарантировать, что каждый `tool_use` получает парный
  непустой `tool_result`, не терять и не обрезать его в ноль.
- Симптом Б, шаг 5 - смягчить формулировки в описании, отдавать пагинацию или
  ручки, чтобы модель не перечитывала ради остатка данных.
