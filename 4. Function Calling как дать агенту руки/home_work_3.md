# Домашнее задание к уроку №4 "Function Calling: как дать агенту руки"

Упражнение 3 (контекст)

## Цель:

Инструмент `run_test` возвращает 4000 строк лога, и к концу сессии агент «глупеет».
Предложите три независимых способа решить проблему, расположив их от самого дешёвого к самому дорогому.
Для каждого укажите минус.

# Решение упражнения 3 (контекст)

## Суть проблемы

4000 строк лога после каждого `run_test` оседают в истории `messages`. Конспект
прямо предупреждает - нельзя скармливать модели гигантские логи целиком. Контекст
забивается мусором, важные ранние сообщения вытесняются, и к концу сессии модель
«глупеет». Ниже три независимых способа, от самого дешёвого к самому дорогому. Они
не исключают друг друга, но каждый решает проблему сам по себе.

## Способ 1 (самый дешёвый). Обрезка и фильтрация в Runtime

Runtime не отдаёт сырые 4000 строк. Он детерминированно сжимает вывод - оставляет
хвост (последние N строк, где обычно итог прогона) плюс строки с маркерами FAIL,
ERROR, Traceback, и схлопывает остальное в счётчик «пропущено M строк». Никаких
внешних вызовов, чистая обработка текста на стороне Runtime.

Минус. Эвристика слепая - значимая строка выше по логу, не попавшая под маркеры,
будет потеряна, и модель не узнает о ней.

## Способ 2 (средний). Ручка на полный лог плюс инструмент догрузки

Runtime сохраняет полный лог в хранилище (файл, объект, запись в БД) и возвращает
модели только ручку - короткую сводку: id лога, сколько тестов прошло и упало,
имена упавших, размер. Рядом добавляется инструмент `get_log_slice(handle, ...)`,
которым модель сама дозапрашивает нужный фрагмент по тесту или диапазону строк.
Это паттерн handles из конспекта - детали доступны по запросу, контекст чистый.

Минус. Лишние round-trip-ы и итерации цикла - модель должна догадаться сделать
дозапрос, и пока она его делает, сессия удлиняется, а Runtime усложняется
хранилищем и временем жизни ручек.

## Способ 3 (самый дорогой). Суммаризация дешёвой моделью

4000 строк прогоняются через дешёвую быструю модель (например Haiku), которая
возвращает структурированную выжимку - что упало, вероятная причина, ключевые
строки. В историю кладётся только эта выжимка. Конспект называет суммаризацию через
дешёвые LLM как отдельный приём управления контекстом.

Минус. Это лишний платный вызов модели и задержка на каждый `run_test`, плюс риск,
что суммаризатор исказит или выкинет деталь, которая была нужна для отладки.

## Почему такой порядок по стоимости

- Способ 1 - чистая функция над текстом, ноль внешних вызовов, дешевле некуда.
- Способ 2 дороже - появляется хранилище и дополнительные итерации агента (лишние
  токены на round-trip-ы), но платим только когда модель реально лезет за деталями.
- Способ 3 самый дорогой - отдельный вызов LLM и его латентность платятся ВСЕГДА,
  даже если лог был зелёный и сжимать особо нечего.

## Код для способов 1 и 2

Самый дешёвый способ и ручка показаны кодом, третий - наброском вызова дешёвой
модели в стиле примеров урока.

```python
"""Управление контекстом для инструмента run_test

Способ 1 - детерминированная обрезка лога в Runtime. Способ 2 - сохранение
полного лога и выдача модели ручки с возможностью догрузки фрагмента
"""

import json
from typing import Any

# маркеры строк, которые нельзя терять при обрезке
FAIL_MARKERS = ("FAIL", "ERROR", "Traceback", "AssertionError")

# сколько строк хвоста оставляем - там обычно итог прогона
TAIL_LINES = 40


def compact_log(log: str, tail_lines: int = TAIL_LINES) -> str:
    """Сжимает лог до значимых строк и хвоста, не трогая исходник

    Оставляет строки с маркерами ошибок и последние tail_lines строк,
    остальное схлопывает в пометку о числе пропущенных строк. Возвращает
    новую строку, вход не меняется
    """
    lines = log.splitlines()
    if len(lines) <= tail_lines:
        return log

    head = lines[:-tail_lines]
    tail = lines[-tail_lines:]
    kept = [ln for ln in head if any(m in ln for m in FAIL_MARKERS)]
    skipped = len(head) - len(kept)

    parts = kept[:]
    if skipped > 0:
        parts.append(f"... пропущено {skipped} строк без ошибок ...")
    parts.extend(tail)
    return "\n".join(parts)


def summarize_counts(log: str) -> dict[str, Any]:
    """Считает короткую сводку прогона для ручки

    Возвращает число строк, признаков падений и первые упавшие маркеры.
    Чистая функция, лог не модифицирует
    """
    lines = log.splitlines()
    failed = [ln for ln in lines if any(m in ln for m in FAIL_MARKERS)]
    return {
        "total_lines": len(lines),
        "fail_lines": len(failed),
        "first_failures": failed[:5],
    }


class LogStore:
    """Хранилище полных логов для выдачи модели ручек (способ 2)"""

    def __init__(self) -> None:
        """Создает пустое хранилище логов в памяти"""
        self._logs: dict[str, str] = {}
        self._counter = 0

    def put(self, log: str) -> str:
        """Сохраняет полный лог и возвращает ручку с краткой сводкой

        В историю модели уйдет только эта ручка, сам лог остается тут
        """
        self._counter += 1
        handle = f"log_{self._counter}"
        self._logs[handle] = log
        summary = summarize_counts(log)
        return json.dumps(
            {"handle": handle, **summary}, ensure_ascii=False
        )

    def get_slice(
        self,
        handle: str,
        start: int = 0,
        count: int = 50,
    ) -> str:
        """Отдает фрагмент сохраненного лога по ручке и диапазону строк

        Если ручка неизвестна - возвращает структурированную ошибку, чтобы
        модель не зависла на несуществующем логе
        """
        log = self._logs.get(handle)
        if log is None:
            return json.dumps(
                {"error_code": "INVALID_ARGUMENT",
                 "message": f"Неизвестная ручка лога: {handle}"},
                ensure_ascii=False,
            )
        chunk = log.splitlines()[start:start + count]
        return "\n".join(chunk)
```

Что показывает код:

1. `compact_log` и `summarize_counts` чистые - вход не меняют, работают только над
   копией строк. Это способ 1 и заготовка сводки для способа 2.
2. `LogStore.put` кладёт полный лог в хранилище и отдаёт модели только ручку со
   сводкой, а `get_slice` даёт безопасную догрузку фрагмента с обработкой
   неизвестной ручки - это способ 2.
3. Способ 3 кодом не дублирую: это просто вызов дешёвой модели (`claude-haiku`) над
   логом перед укладкой результата в историю, по аналогии с примерами урока.
