# Кейс 2. Агент аналитики данных

## 🧩 проблема:

доступ к данным осуществляется через ручные SQL-запросы, что замедляет анализ;

## ⚙️ почему нужен агент:

требуется упростить получение данных и сократить время на аналитику;

## 🤖 что он должен делать:

принимать текстовый запрос;
формировать SQL-запрос;
выполнять запрос к базе данных;
объяснять результат;

## 🔒 ограничения:

ограниченный доступ к данным;
необходимость безопасного выполнения SQL-запросов;

## ⚠️ сложности:

неоднозначные формулировки запросов;
сложные связи между таблицами;

---

# Решение домашнего задания

## 1. Прикладной сценарий

Агент-аналитик (text-to-SQL). На вход приходит вопрос на естественном языке
("сколько новых клиентов пришло в прошлом месяце", "топ-10 товаров по выручке за квартал").

### Агент должен:

- понять вопрос и сопоставить его со схемой данных
- сформировать корректный и безопасный SQL-запрос
- выполнить его в БД (только на чтение)
- объяснить результат человеческим языком, при необходимости подсказать визуализацию

### Ключевые ограничения:

Безопасность и доступ к данным. Агент работает под read-only учёткой, видит только разрешённые таблицы/витрины, не может
изменять или удалять данные. Любой потенциально опасный или дорогой запрос проходит проверку до выполнения.

### Граница ответственности:

Простые и средней сложности выборки агент строит и выполняет сам, очень сложные/неоднозначные - уточняет у пользователя
или показывает черновик SQL на подтверждение.

## 2. Цикл работы агента Reason -> Act -> Observe

```
                    +------------------------+
   вопрос  ----->   |  Подготовка контекста  |
                    |  (схема БД, описания   |
                    |  таблиц, примеры SQL)  |
                    +-----------+------------+
                                |
                                v
            +-------------------------------------------+
            |                  REASON                   |
            |  понять намерение, выбрать таблицы,       |
            |  спланировать запрос, оценить сложность,  |
            |  выбрать модель                           |
            +-------------------+-----------------------+
                                |
                                v
            +-------------------------------------------+
            |                   ACT                     |
            |  - сгенерировать SQL                       |
            |  - прогнать через валидатор/линтер         |
            |  - EXPLAIN (план запроса, оценка стоимости)|
            |  - выполнить (read-only, LIMIT, таймаут)   |
            |  - либо уточнить вопрос у пользователя     |
            +-------------------+-----------------------+
                                |
                                v
            +-------------------------------------------+
            |                 OBSERVE                   |
            |  - выполнился ли запрос без ошибок          |
            |  - разумен ли результат (не пусто/не млн)   |
            |  - не зациклились ли (счётчик попыток)      |
            +-------------------+-----------------------+
                                |
              ок?               |     ошибка SQL / странный результат
            +-------------------+-------------------+
            |                                       |
            v                                       v
   +------------------+                  +----------------------+
   | Ответ + объясн.  |                  | новый виток:         |
   | + сам SQL        |                  | починить SQL по      |
   +------------------+                  | тексту ошибки или    |
                                         | уточнить у юзера     |
                                         +----------------------+
```

### Подзадачи:

1. Понимание вопроса и сопоставление со схемой
2. Планирование запроса (какие таблицы, join, агрегаты, фильтры)
3. Генерация SQL
4. Статическая проверка (только SELECT, разрешённые таблицы, есть LIMIT)
5. EXPLAIN и оценка стоимости плана до выполнения
6. Выполнение и интерпретация результата
7. Починка по тексту ошибки БД (self-repair цикл)

## 3. Этапы, где нужны разные модели

| Этап                                     | Тип модели                  | Почему                                            |
|------------------------------------------|-----------------------------|---------------------------------------------------|
| Классификация сложности вопроса          | дешёвая (Haiku 4.5)         | быстро понять, простая выборка или сложная        |
| Генерация простого SQL                   | дешёвая/средняя             | один-два join, типовая агрегация                  |
| Генерация сложного SQL                   | продвинутая (Opus 4.8)      | оконные функции, CTE, много таблиц, хитрая логика |
| Починка SQL по ошибке БД                 | средняя (Sonnet 4.6)        | достаточно для исправления синтаксиса/типов       |
| Объяснение результата                    | средняя (Sonnet 4.6)        | связный текст по табличке - не требует Opus       |
| Поиск нужных таблиц (если схема большая) | embeddings + дешёвая модель | RAG по описаниям таблиц/колонок                   |

Если схема БД большая (сотни таблиц), её целиком в промпт не положить - нужен retrieval по описаниям таблиц и колонок
(эмбеддинги + векторный поиск), в контекст попадают только релевантные таблицы.

## 4. Логика роутинга моделей (data-driven)

```
вопрос
  |
  v
[Haiku 4.5: оценка сложности + выбор релевантных таблиц]
  |
  +-- простая выборка (1-2 таблицы, типовая агрегация)
  |        -> Sonnet 4.6 (или даже Haiku) генерирует SQL
  |
  +-- сложный запрос (оконные функции, CTE, 3+ join)
  |        -> Opus 4.8
  |
  +-- неоднозначный вопрос / не нашли таблицы
           -> уточнение у пользователя, без генерации
```

### Сигналы для роутинга:

- метка сложности от классификатора
- число задействованных таблиц и тип агрегации
- оценка стоимости плана из EXPLAIN (дорогой план - повод показать SQL человеку)
- история неудачных попыток (после N ошибок - эскалация на сложную модель или к
  пользователю)

### Data-driven часть:

Пороги сложности и распределение по моделям калибруем на наборе пар "вопрос -> эталонный SQL"
(свой held-out набор или прогон по бенчмарку типа Spider/BIRD).
Метрики - доля корректных запросов (execution accuracy), доля запросов с ошибкой выполнения, средняя стоимость на
вопрос.

## 5. Защита от Dead Loop

- **счётчик попыток починки SQL** - лимит на self-repair (например 3 попытки), потом
  показываем пользователю последнюю ошибку и черновик SQL
- **бюджет токенов и таймаут** на обработку вопроса
- **детектор повторов** - если модель генерирует один и тот же неработающий SQL, выходим из цикла
- **таймаут на сам SQL в БД** - длинный запрос убивается на стороне БД (statement timeout)
- **критерий прогресса** - каждая попытка должна давать новый SQL или новую диагностику, иначе тупик
- **fallback** - не смогли построить корректный запрос, отдаём пользователю черновик SQL и просим уточнить или поправить

## 6. Лимиты провайдера и где агент свалится

Лимиты LLM-провайдера (Anthropic): RPM, токены в минуту (вход/выход), токены в сутки,
контекстное окно (Opus 4.8 / Sonnet 4.6 - до 1M, Haiku 4.5 - 200K).
Отдельно лимиты самой БД (число коннектов, нагрузка).

| Где падает                                   | Что делаем                                                |
|----------------------------------------------|-----------------------------------------------------------|
| 429 / 5xx от LLM                             | ретраи с backoff, при затяжке - откат на дешёвую модель   |
| синтаксическая ошибка SQL                    | self-repair по тексту ошибки (ограниченное число попыток) |
| опасный SQL (не SELECT, запрещённая таблица) | блокируем до выполнения статической проверкой             |
| слишком тяжёлый план (EXPLAIN)               | не выполняем, просим уточнить или подтвердить             |
| таймаут запроса в БД                         | убиваем по statement timeout, сообщаем пользователю       |
| пустой/абсурдный результат                   | помечаем, предлагаем переформулировать                    |
| схема не влезает в контекст                  | RAG по таблицам, в контекст только релевантные            |

### Безопасность:

Отдельный приоритет. Агент ходит в БД только под read-only ролью, SQL проходит через белый список
(только SELECT, только разрешённые таблицы/схемы), на каждый запрос навешивается LIMIT и statement timeout. Это
защищает и от инъекций, и от случайно тяжёлых запросов.

## 7. Системный промпт - что в нём лежит

- **инструкции агента** - роль, правила (только SELECT, обязательный LIMIT, объяснять результат,
  при неоднозначности уточнять) - ~1000-1500 токенов
- **схема данных** - DDL/описания релевантных таблиц и колонок, связи между ними - переменная часть,
  подтягивается под запрос
- **few-shot примеры** пар "вопрос -> правильный SQL" по этой БД
- **бизнес-глоссарий** - что значит "активный клиент", "выручка" и т.п. (важно для снятия неоднозначности)
- **история диалога** при уточнениях

Стабильную часть (инструкции, глоссарий, базовые примеры) кешируем через prompt caching.
Релевантные таблицы под конкретный вопрос идут после точки кеширования.

## 8. Прикидка экономики

Цены Claude (за 1 млн токенов, вход / выход):

| Модель            | Вход  | Выход  |
|-------------------|-------|--------|
| Claude Haiku 4.5  | $1.00 | $5.00  |
| Claude Sonnet 4.6 | $3.00 | $15.00 |
| Claude Opus 4.8   | $5.00 | $25.00 |

Стоимость одного типового вопроса (грубо, без кеша):

1. Классификатор + выбор таблиц (Haiku): вход ~1500, выход ~100
    - 1500/1e6 * $1 + 100/1e6 * $5 ~ $0.002
2. Генерация SQL (Sonnet): вход ~3000 (схема + примеры), выход ~300
    - 3000/1e6 * $3 + 300/1e6 * $15 = $0.009 + $0.0045 ~ $0.0135
3. Объяснение результата (Sonnet): вход ~1500, выход ~300 ~ $0.009

Итого порядка $0.025 за типовой вопрос. Сложный запрос на Opus 4.8 c парой попыток
починки выйдет дороже - примерно $0.06-0.10.

С prompt caching на стабильной части (инструкции + глоссарий + базовые примеры) вход по кешу стоит ~0.1x от обычного,
реальная цена падает в разы. Запись в кэш при этом дороже обычного входа примерно в 1.25 раза (для TTL 5 минут),
так что выгода начинается со второго запроса с тем же префиксом.

Оценка при 5 000 вопросов в месяц и среднем чеке $0.025:

- без кеша: ~$125 / мес
- с кешированием: ориентировочно $50-80 / мес (порядка 5-7 тыс. руб.)

Дешевле, чем держать аналитика на ручные ad-hoc выборки. Рычаги экономии: роутинг (дешёвая модель на простое),
prompt caching схемы и примеров, RAG по таблицам вместо всей схемы в контексте, лимит попыток self-repair.

## 9. Итоговая архитектура (кратко)

```
Пользователь (вопрос на естественном языке)
    |
    v
[Оркестратор агента] -- цикл Reason/Act/Observe, лимиты, self-repair
    |          |               |              |
    v          v               v              v
[Классиф.] [Поиск таблиц] [Генерация SQL] [Объяснение]
 Haiku 4.5  emb + RAG      Sonnet/Opus      Sonnet 4.6
                |                |
                v                v
        [Описания схемы]  [Валидатор SQL: только SELECT,
                           whitelist, LIMIT] -> [EXPLAIN]
                                |
                                v
                    [Read-only подключение к БД,
                     statement timeout]
    |
    v
[Логи запросов, метрики точности, накопление пар вопрос->SQL]
```

### Принцип:

Дешёвая модель оценивает сложность и находит таблицы, средняя строит типовой SQL, продвинутая включается на сложных
запросах, а каждый сгенерированный SQL проходит проверку безопасности и EXPLAIN до выполнения под read-only учёткой.
