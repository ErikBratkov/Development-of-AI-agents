# Кейс 4. Агент работы с кодом

## 🧩 проблема:

разработчики тратят много времени на написание и анализ кода;

## ⚙️ почему нужен агент:

требуется ускорить разработку и снизить количество рутинных задач;

## 🤖 что он должен делать:

принимать задачу или фрагмент кода;
генерировать или дорабатывать код;
объяснять логику решения;

## 🔒 ограничения:

ограниченный контекст проекта;
требования к корректности и читаемости кода;

## ⚠️ сложности:

неоднозначные требования к задаче;
риск генерации некорректного или небезопасного кода.

---

# Решение домашнего задания

## 1. Прикладной сценарий

Агент работы с кодом. На вход приходит задача ("добавь валидацию входных данных в этот обработчик",
"почему падает этот тест", "напиши функцию парсинга конфига") или фрагмент кода.

### Агент должен:

- понять задачу и собрать нужный контекст проекта
- сгенерировать или доработать код в стиле проекта
- объяснить логику решения
- по возможности проверить себя (линтер, тесты)

### Ключевые ограничения:

Корректность, читаемость и безопасность кода, при
ограниченном контексте проекта (весь репозиторий в окно не влезает). Поэтому нужен
точечный сбор контекста - поиск по коду, чтение только релевантных файлов.

### Граница ответственности:

генерацию и правки кода агент делает сам и прогоняет проверки, но изменения уходят разработчику на ревью - агент
не коммитит и не деплоит без подтверждения.

## 2. Цикл работы агента Reason -> Act -> Observe

```
                    +------------------------+
  задача   ----->   |  Подготовка контекста  |
                    |  (релевантные файлы,   |
                    |  стандарты проекта,    |
                    |  структура репо)       |
                    +-----------+------------+
                                |
                                v
            +--------------------------------------------+
            |                  REASON                    |
            |  понять требование, найти где менять,      |
            |  спланировать изменение, оценить сложность,|
            |  выбрать модель                            |
            +-------------------+------------------------+
                                |
                                v
            +--------------------------------------------+
            |                   ACT                      |
            |  - поиск/чтение файлов (grep, read)        |
            |  - генерация или правка кода (edit)        |
            |  - запуск линтера / тестов                 |
            +-------------------+------------------------+
                                |
                                v
            +---------------------------------------------+
            |                 OBSERVE                     |
            |  - компилируется / проходят ли тесты        |
            |  - чистый ли линтер                         |
            |  - решена ли исходная задача                |
            |  - счётчик итераций                         |
            +-------------------+-------------------------+
                                |
              ок?               |     ошибки сборки / падают тесты
            +-------------------+-------------------+
            |                                       |
            v                                       v
   +------------------+                  +----------------------+
   | Готовый diff     |                  | новый виток:         |
   | + объяснение     |                  | починить по выводу   |
   | -> на ревью      |                  | компилятора/тестов   |
   +------------------+                  +----------------------+
```

### Подзадачи:

1. Понимание требования и сбор контекста (поиск по коду, чтение файлов)
2. Планирование изменения (что и где менять)
3. Генерация или правка кода в стиле проекта
4. Запуск проверок (компиляция, линтер, тесты)
5. Самопочинка по выводу ошибок (self-repair цикл)
6. Объяснение решения и подготовка diff на ревью

## 3. Этапы, где нужны разные модели

| Этап                                 | Тип модели                  | Почему                                              |
|--------------------------------------|-----------------------------|-----------------------------------------------------|
| Классификация задачи                 | дешёвая (Haiku 4.5)         | понять - простая правка или сложная разработка      |
| Поиск релевантных файлов             | embeddings + дешёвая модель | RAG по кодовой базе, в контекст только нужное       |
| Простая правка / шаблонный код       | средняя (Sonnet 4.6)        | хороший баланс на типовых задачах                   |
| Сложная разработка / отладка         | продвинутая (Opus 4.8)      | многофайловые изменения, нетривиальная логика, баги |
| Починка по ошибке компилятора/тестов | средняя (Sonnet 4.6)        | достаточно для исправления по диагностике           |
| Объяснение решения                   | дешёвая/средняя             | связный текст по готовому коду                      |

Контекст проекта в окно не влезает целиком, поэтому собираем точечно - поиск по коду (grep/символьный поиск) и
semantic search по эмбеддингам, читаем только релевантные файлы. Это и есть работа с ограниченным контекстом.

## 4. Логика роутинга моделей (data-driven)

```
задача
  |
  v
[Haiku 4.5: оценка сложности + поиск где менять]
  |
  +-- простая правка (один файл, понятное изменение)
  |        -> Sonnet 4.6
  |
  +-- сложная разработка / отладка (много файлов, баг)
  |        -> Opus 4.8
  |
  +-- неоднозначное требование
           -> уточнение у пользователя до генерации
```

### Сигналы для роутинга:

- метка сложности от классификатора
- число затронутых файлов и размер изменения
- наличие отладки (надо понять причину бага, а не просто написать код)
- история неудачных попыток (после N провалов тестов - на Opus 4.8)

### Data-driven часть:

Пороги и распределение по моделям калибруем на наборе задач с эталонными решениями или на бенчмарке
(например SWE-bench для реальных багфиксов, HumanEval для генерации функций). Метрики - доля задач,
прошедших тесты (pass rate), доля изменений, принятых на ревью без правок, средняя стоимость на задачу.

## 5. Защита от Dead Loop

- **счетчик итераций self-repair** - лимит на число попыток починить код по
  ошибкам (например 3-4), потом отдаём как есть с описанием проблемы
- **бюджет токенов и таймаут** на задачу
- **детектор повторов** - если модель выдаёт один и тот же неработающий патч или
  ходит по кругу между двумя вариантами, выходим
- **критерий прогресса** - число падающих тестов/ошибок должно уменьшаться, иначе стоп
- **таймаут на запуск тестов/сборки** - чтобы не зависнуть на долгом прогоне
- **fallback** - не довели до зелёных тестов, отдаём разработчику текущий diff, логи ошибок и описание, где застряли

## 6. Лимиты провайдера и где агент свалится

### Лимиты LLM-провайдера:

RPM, токены в минуту/сутки, контекстное окно (Opus 4.8 / Sonnet 4.6 - до 1M, Haiku 4.5 - 200K).
Большое окно тут особенно ценно - можно держать много файлов в контексте, но всё равно не весь репозиторий.

| Где падает                   | Что делаем                                                       |
|------------------------------|------------------------------------------------------------------|
| 429 / 5xx от LLM             | ретраи с backoff, откат на дешёвую модель                        |
| проект не влезает в контекст | RAG/поиск по коду, читаем только релевантные файлы               |
| код не компилируется         | self-repair по выводу компилятора (лимит попыток)                |
| падают тесты                 | анализ вывода тестов и правка (ограниченное число попыток)       |
| неоднозначное требование     | уточнение у пользователя до генерации                            |
| риск небезопасного кода      | проверка линтером/сканером, опасные паттерны - на ревью человеку |
| долгий прогон тестов/сборки  | таймаут, частичная проверка                                      |

### Безопасность:

Агент не выполняет произвольные команды без ограничений, прогон тестов идёт в изолированной среде (контейнер/sandbox),
сгенерированный код проходит линтер и статический анализ. Секреты и креды не кладутся в контекст и не коммитятся.
Любое изменение - через ревью, без автодеплоя.

## 7. Системный промпт - что в нём лежит

- **инструкции агента** - роль, правила (стиль проекта, читаемость, не выдумывать
  API, проверять себя тестами, при неоднозначности уточнять) - ~1500 токенов
- **стандарты кодирования проекта** - PEP8, линтер-конфиг, требования к docstring и
  аннотациям типов, конвенции именования (стабильная часть)
- **релевантные файлы и фрагменты кода** - переменная часть, подтягивается под задачу через поиск
- **структура репозитория / карта модулей** - чтобы агент понимал, где что лежит
- **few-shot примеры** хорошего кода и правок в стиле проекта
- **история диалога и предыдущие правки** - переменная часть

Cтабильную часть (инструкции, стандарты, примеры) кешируем через prompt caching.
Релевантные файлы под конкретную задачу идут после точки кеширования.
Кодовую базу целиком в промпт не кладём - только то, что нашли поиском.

## 8. Прикидка экономики

Цены Claude (за 1 млн токенов, вход / выход):

| Модель            | Вход  | Выход  |
|-------------------|-------|--------|
| Claude Haiku 4.5  | $1.00 | $5.00  |
| Claude Sonnet 4.6 | $3.00 | $15.00 |
| Claude Opus 4.8   | $5.00 | $25.00 |

Задачи с кодом контекстно-тяжёлые (много файлов на входе), поэтому вход обычно большой.
Типовая правка (грубо, без кеша):

1. Классификатор + поиск файлов (Haiku): вход ~2000, выход ~150 ~ $0.0028
2. Генерация правки (Sonnet): вход ~8000 (несколько файлов + стандарты),
   выход ~800
    - 8000/1e6 * $3 + 800/1e6 * $15 = $0.024 + $0.012 ~ $0.036
3. Починка по тестам (Sonnet, 1-2 итерации): вход ~6000, выход ~500 каждая
    - на итерацию: 6000/1e6 * $3 + 500/1e6 * $15 = $0.018 + $0.0075 ~ $0.0255
    - две итерации ~ $0.051

Итого порядка $0.09 за типовую правку. Сложная задача на Opus 4.8 с несколькими
файлами и отладкой выйдет существенно дороже - примерно $0.30-0.60 (вход на Opus
дороже, и итераций больше).

Prompt caching тут даёт особенно сильный эффект: стабильные стандарты проекта и
часто читаемые файлы кешируются, и при многоитерационной починке вход по кешу
стоит ~0.1x. Запись в кэш дороже обычного входа примерно в 1.25 раза (TTL 5 минут),
но при нескольких итерациях это окупается с запасом. Итого кэш может срезать
стоимость в 2-4 раза.

Оценка при 4 000 задач в месяц и среднем чеке $0.10:

- без кеша: ~$400 / мес
- с кешированием: ориентировочно $150-250 / мес (порядка 14-22 тыс. руб.)

Дешевле, чем время разработчиков на рутину. Рычаги экономии: роутинг (Sonnet на
простое, Opus только на сложное), prompt caching стандартов и горячих файлов,
RAG/поиск вместо всей кодовой базы в контексте, лимит итераций self-repair.

## 9. Итоговая архитектура (кратко)

```
Пользователь (задача / фрагмент кода)
    |
    v
[Оркестратор агента] -- цикл Reason/Act/Observe, лимиты, self-repair
    |          |               |               |
    v          v               v               v
[Классиф.] [Поиск по коду]  [Генерация/правка] [Объяснение]
 Haiku 4.5  grep + emb RAG    Sonnet/Opus        Sonnet
                |                 |
                v                 v
        [Файлы проекта]    [Линтер + статический анализ]
                                  |
                                  v
                        [Прогон тестов в sandbox]
                                  |
                                  v
                          [diff -> ревью разработчика]
    |
    v
[Логи, метрики pass rate, накопление эталонных решений]
```

### Принцип:

Дешёвая модель оценивает сложность и находит, где менять, средняя делает типовые правки,
продвинутая включается на сложной разработке и отладке, а каждое изменение проходит проверки (линтер, тесты в sandbox)
и уходит разработчику на ревью - без автокоммита и автодеплоя.
