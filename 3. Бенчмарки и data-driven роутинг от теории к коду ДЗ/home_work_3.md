# Кейс 3. Агент автоматизации задач

## 🧩 проблема:

операции в сервисах выполняются вручную и занимают время;

## ⚙️ почему нужен агент:

требуется автоматизировать выполнение действий через API;

## 🤖 что он должен делать:

принимать команду пользователя;
разбивать задачу на шаги;
вызывать API внешнего сервиса;
контролировать результат выполнения;

## 🔒 ограничения:

лимиты API и нестабильность внешних сервисов;
ограничения доступа по токенам;

## ⚠️ сложности:

ошибки интеграции и таймауты;
необходимость повторных попыток выполнения;

---

# Решение домашнего задания

## 1. Прикладной сценарий

Агент автоматизации задач (оркестратор API). На вход приходит команда на естественном языке
("создай задачу в трекере и оповести команду в чате", "выгрузи отчёт и положи его в облако").

### Агент должен:

- разобрать команду и составить план из шагов
- по каждому шагу вызвать нужный внешний API
- контролировать результат каждого вызова, обрабатывать ошибки и таймауты
- при сбое - повторить, откатить или эскалировать

### Ключевая особенность:

Aгент совершает реальные действия с побочными эффектами (создаёт сущности, шлёт сообщения).
Поэтому критичны идемпотентность, повторы, откат и подтверждение опасных операций.
Деструктивные и необратимые действия (удаление, отправка наружу, платёж) требуют подтверждения.

### Граница ответственности:

Безопасные и обратимые шаги агент выполняет сам, опасные и необратимые - только после подтверждения
человеком (human-in-the-loop).

## 2. Цикл работы агента Reason -> Act -> Observe

```
                    +------------------------+
  команда  ----->   |  Подготовка контекста  |
                    |  (описания доступных   |
                    |  инструментов/API,     |
                    |  права токена)         |
                    +-----------+------------+
                                |
                                v
            +-------------------------------------------+
            |                  REASON                   |
            |  разбить задачу на шаги, выбрать API/tool  |
            |  для текущего шага, оценить риск действия, |
            |  выбрать модель                            |
            +-------------------+-----------------------+
                                |
                                v
            +-------------------------------------------+
            |                   ACT                     |
            |  - вызов внешнего API (tool call)          |
            |  - для опасных действий - запрос           |
            |    подтверждения у пользователя            |
            +-------------------+-----------------------+
                                |
                                v
            +-------------------------------------------+
            |                 OBSERVE                   |
            |  - успешен ли вызов (код ответа, payload)  |
            |  - не таймаут / не rate limit              |
            |  - выполнен ли весь план                   |
            |  - счётчик шагов и повторов                |
            +-------------------+-----------------------+
                                |
        шаг ок и план не завершён|     ошибка/таймаут        план завершён
            +-------------------+--------------+----------------+
            |                                  |                |
            v                                  v                v
   +------------------+              +------------------+  +--------------+
   | следующий шаг    |              | ретрай с backoff |  | Финальный    |
   | плана (Reason)   |              | / откат / эскал. |  | отчёт юзеру  |
   +------------------+              +------------------+  +--------------+
```

### Подзадачи:

1. Декомпозиция команды на упорядоченный план шагов
2. Выбор инструмента/API под каждый шаг (function calling / tool use)
3. Оценка риска действия (обратимое/необратимое)
4. Вызов API с корректными параметрами
5. Контроль результата и обработка ошибок (ретраи, backoff)
6. Откат уже выполненных шагов при неустранимом сбое (сага/компенсации)
7. Сбор итогового отчёта о том, что сделано

## 3. Этапы, где нужны разные модели

| Этап                                    | Тип модели             | Почему                                      |
|-----------------------------------------|------------------------|---------------------------------------------|
| Классификация команды / простой маршрут | дешёвая (Haiku 4.5)    | один-два понятных действия                  |
| Планирование сложной задачи             | продвинутая (Opus 4.8) | многошаговый план с зависимостями шагов     |
| Выбор tool и сборка аргументов          | средняя (Sonnet 4.6)   | надёжный function calling по схеме          |
| Разбор ошибки API и решение что делать  | средняя (Sonnet 4.6)   | понять, ретраить, откатить или эскалировать |
| Формирование отчёта пользователю        | дешёвая/средняя        | связный текст по логу выполнения            |

Сами вызовы API - это инструменты (tools), которые исполняет наш код (harness), а не модель.
Модель только решает, какой инструмент вызвать и с какими аргументами.
Опасные инструменты помечаются как требующие подтверждения.

## 4. Логика роутинга моделей (data-driven)

```
команда
  |
  v
[Haiku 4.5: классификатор - простая команда или сложная задача]
  |
  +-- простая (1-2 шага, понятные API)
  |        -> Sonnet 4.6 планирует и вызывает tools
  |
  +-- сложная (много шагов, зависимости, ветвления)
  |        -> Opus 4.8 строит план, дальше шаги исполняет Sonnet
  |
  +-- неоднозначная / опасная команда
           -> уточнение или запрос подтверждения у пользователя
```

### Сигналы для роутинга:

- метка сложности от классификатора
- число шагов в плане и наличие зависимостей между ними
- уровень риска действий (есть ли необратимые операции)
- частота ошибок интеграции на этом сценарии (по логам)

### Data-driven часть:

Распределение по моделям и пороги подбираем на наборе реальных команд с эталонными планами.
Метрики - доля успешно доведённых до конца задач (task success rate), доля корректно выбранных API-вызовов,
среднее число ретраев, стоимость на задачу. Сравнение моделей по function calling можно вести
на бенчмарке вроде BFCL (Berkeley Function-Calling Leaderboard).

## 5. Защита от Dead Loop

Тут зацикливание особенно опасно - агент может бесконечно дёргать внешний API.

- **счётчик шагов плана** - жёсткий лимит на число действий в одной задаче
- **счётчик ретраев на шаг** - ограниченное число повторов с экспоненциальной задержкой, потом откат/эскалация
- **детектор повторов** - один и тот же вызов с теми же аргументами и тем же
  результатом несколько раз подряд = тупик, выходим
- **бюджет токенов и общий таймаут** на задачу
- **идемпотентность** - повторные вызовы по возможности идемпотентны (idempotency key), чтобы ретрай не создавал дубли
- **критерий прогресса** - каждый шаг должен менять состояние задачи, иначе стоп
- **fallback** - не довели задачу автоматически - откатываем сделанное и отдаём
  человеку с подробным логом

## 6. Лимиты провайдера и где агент свалится

Два слоя лимитов: LLM-провайдер (RPM, токены в минуту/сутки, контекст - Opus 4.8 /
Sonnet 4.6 до 1M, Haiku 4.5 200K) и внешние сервисы (их собственные rate limits,
квоты, права токена).

| Где падает                      | Что делаем                                         |
|---------------------------------|----------------------------------------------------|
| 429 / 5xx от LLM                | ретраи с backoff, откат на дешёвую модель          |
| 429 от внешнего API             | уважаем Retry-After, backoff, очередь, троттлинг   |
| таймаут внешнего API            | повтор с backoff, после лимита - откат/эскалация   |
| 401/403 (истёк/нет прав токена) | обновляем токен (refresh), если нельзя - эскалация |
| частичное выполнение плана      | компенсирующие действия (откат уже созданного)     |
| необратимое действие            | подтверждение у пользователя до выполнения         |
| нестабильный сервис (флапает)   | circuit breaker - временно перестаём дёргать       |

Токены и секреты внешних сервисов не кладём в промпт и не показываем модели - они живут на стороне harness/хранилища
секретов и подставляются при вызове. Это защищает от утечки через историю диалога и от prompt injection.

## 7. Системный промпт - что в нём лежит

- **инструкции агента** - роль, правила (подтверждать опасные действия, ретраить
  по правилам, при неоднозначности уточнять, не выдумывать API) - ~1500 токенов
- **описания доступных инструментов/API** - имена, назначение, схемы аргументов,
  пометка опасных - стабильная часть (при большом числе API - tool search, подгружаем схемы по релевантности)
- **политики безопасности** - что считается необратимым, что требует подтверждения
- **few-shot примеры** хороших планов и корректной обработки ошибок
- **состояние текущей задачи и лог выполненных шагов** - переменная часть

Стабильную часть (инструкции, политики, описания инструментов) кешируем. Переменное состояние задачи идёт после
точки кеширования. Если инструментов много - tool search, чтобы схемы добавлялись, а не переписывали префикс
(кеш не ломается).

## 8. Прикидка экономики

Цены Claude (за 1 млн токенов, вход / выход):

| Модель            | Вход  | Выход  |
|-------------------|-------|--------|
| Claude Haiku 4.5  | $1.00 | $5.00  |
| Claude Sonnet 4.6 | $3.00 | $15.00 |
| Claude Opus 4.8   | $5.00 | $25.00 |

Задача обычно многошаговая, поэтому считаем на задачу, а не на один вызов модели.
Типовая задача из 3 шагов (грубо, без кеша):

1. Классификатор (Haiku): вход ~1200, выход ~80 ~ $0.0016
2. Планирование (Sonnet): вход ~3000 (инструкции + описания API), выход ~400
    - 3000/1e6 * $3 + 400/1e6 * $15 ~ $0.015
3. 3 шага tool call (Sonnet, каждый вход ~2500, выход ~200)
    - на шаг: 2500/1e6 * $3 + 200/1e6 * $15 = $0.0075 + $0.003 ~ $0.0105
    - три шага ~ $0.0315
4. Отчёт (Sonnet): вход ~1500, выход ~250 ~ $0.008

Итого порядка $0.056 за типовую задачу. Сложная задача с планированием на Opus 4.8
и повторами выйдет дороже - примерно $0.12-0.20.

С prompt caching на стабильной части (инструкции + описания API + политики) цена
заметно падает, особенно когда в одной задаче несколько вызовов с одним префиксом.
Чтение из кэша стоит ~0.1x от цены входа, запись - примерно 1.25x (TTL 5 минут),
поэтому кэш окупается уже со второго вызова.

Оценка при 3 000 задач в месяц и среднем чеке $0.06:

- без кеша: ~$180 / мес
- с кешированием: ориентировочно $80-120 / мес (порядка 7-11 тыс. руб.)

### Рычаги экономии:

роутинг (Haiku/Sonnet на простое, Opus только на планирование сложного), prompt caching префикса с описаниями
API, tool search вместо всех схем сразу, ограничение шагов и ретраев.

## 9. Итоговая архитектура (кратко)

```
Пользователь (команда)
    |
    v
[Оркестратор агента] -- цикл Reason/Act/Observe, лимиты, ретраи, откаты
    |          |              |                  |
    v          v              v                  v
[Классиф.] [Планировщик]  [Исполнитель шагов]  [Отчёт]
 Haiku 4.5   Opus 4.8       Sonnet 4.6 +         Sonnet
                            function calling
                                |
              +-----------------+------------------+
              v                 v                  v
        [Внешний API 1]   [Внешний API 2]    [подтверждение
         retry/backoff     idempotency key     опасных действий]
         circuit breaker
                                |
                                v
                      [Хранилище секретов/токенов]
    |
    v
[Логи выполнения, метрики task success, накопление эталонных планов]
```

### Принцип:

дешёвая модель маршрутизирует, продвинутая планирует сложное, средняя надёжно вызывает API через
function calling, а каждое действие проходит контроль - ретраи с backoff, идемпотентность, откат при сбое и
подтверждение для необратимых операций. Секреты остаются вне модели.
