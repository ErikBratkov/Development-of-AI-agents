# Кейс 1. Агент поддержки

## 🧩 проблема:

входящие пользовательские запросы обрабатываются вручную через тикет-систему и требуют много времени;

## ⚙️ почему нужен агент:

требуется снизить нагрузку на поддержку и ускорить обработку типовых запросов;

## 🤖 что он должен делать:

принимать текст запроса пользователя;
искать релевантную информацию в базе знаний (документы/FAQ);
формировать ответ с учетом найденного контекста;

## 🔒 ограничения:

ограниченный набор документов;
требования к точности и проверяемости ответа;

## ⚠️ сложности:

неструктурированные и неоднозначные запросы;
отсутствие или противоречивость информации в базе;

---

# Решение домашнего задания

## 1. Прикладной сценарий

Агент первой линии поддержки. На вход приходит текст обращения пользователя
(чат, почта, форма тикет-системы).

### Агент должен:

- понять суть запроса и его сложность
- найти релевантные фрагменты в базе знаний (документы, FAQ, регламенты)
- сформировать ответ строго на основе найденного контекста с указанием источников
- если ответа в базе нет или данные противоречивы - честно сказать об этом и
  передать обращение живому оператору (эскалация)

### Ключевое ограничение:

Проверяемость. Агент не выдумывает, а отвечает только тем, что подтверждено документами. Каждый ответ сопровождается
ссылками на источники.

### Граница ответственности:

Типовые и средней сложности запросы агент закрывает сам, сложные/спорные/потенциально дорогие по последствиям - уходят
человеку.

### Требование к скорости ответа:

Первая фраза ответа должна начать генерироваться не дольше чем через 500 мсек после получения
обращения - пользователь не должен ждать перед пустым экраном. Как это выполняется - в разделе 9.

## 2. Цикл работы агента Reason -> Act -> Observe

Агент работает итеративно в цикле, пока не получит готовый проверенный ответ либо не упрётся в лимит шагов.

```
                    +------------------------+
   запрос  ----->   |   Подготовка контекста |
                    | (системный промпт,     |
                    |  история диалога)      |
                    +-----------+------------+
                                |
                                v
            +-------------------------------------------+
            |                  REASON                   |
            |  классификация запроса, выбор действия    |
            |  (искать в базе / уточнить / ответить /   |
            |   эскалировать), выбор модели по сложности|
            +-------------------+-----------------------+
                                |
                                v
            +-------------------------------------------+
            |                   ACT                     |
            |  выполнение выбранного действия:          |
            |  - retrieval по базе знаний               |
            |  - запрос уточнения у пользователя        |
            |  - генерация ответа на основе контекста   |
            |  - эскалация на оператора                 |
            +-------------------+-----------------------+
                                |
                                v
            +-------------------------------------------+
            |                 OBSERVE                   |
            |  оценка результата:                       |
            |  - релевантны ли найденные документы      |
            |  - достаточно ли данных для ответа        |
            |  - нет ли противоречий                    |
            |  - не зациклились ли мы (счётчик шагов)   |
            +-------------------+-----------------------+
                                |
              достаточно?       |     не достаточно
            +-------------------+-------------------+
            |                                       |
            v                                       v
   +------------------+                  +----------------------+
   |  Финальный ответ |                  |  новый виток цикла   |
   |  + источники     |                  |  (Reason) или        |
   +------------------+                  |  эскалация по лимиту |
                                         +----------------------+
```

### Подзадачи внутри цикла:

1. Классификация и роутинг запроса (простой/сложный, тема, язык)
2. Извлечение из базы знаний (embedding + векторный поиск, при необходимости
   реранкинг)
3. Проверка достаточности и непротиворечивости найденного
4. Генерация ответа с цитированием источников
5. Самопроверка ответа (нет ли выдумок, всё ли опирается на контекст)
6. Принятие решения - отдать ответ или эскалировать

## 3. Этапы, где нужны разные модели

Не все шаги требуют самой умной (и самой дорогой) модели. Разделяем нагрузку:

| Этап                              | Тип модели                   | Почему                                                        |
|-----------------------------------|------------------------------|---------------------------------------------------------------|
| Классификация и роутинг           | дешёвая (Haiku 4.5)          | простая задача, нужна скорость и низкая цена                  |
| Векторизация запроса (embeddings) | отдельная embedding-модель * | у Anthropic своих эмбеддингов нет, берём Voyage AI или аналог |
| Реранкинг найденных фрагментов    | дешёвая модель / rerank API  | быстрое сравнение, дорогая модель тут избыточна               |
| Ответ на типовой FAQ              | средняя (Sonnet 4.6)         | хороший баланс качества и стоимости                           |
| Синтез ответа при противоречиях   | продвинутая (Opus 4.8)       | нужно рассуждение, сопоставление источников                   |
| Самопроверка финального ответа    | средняя (Sonnet 4.6)         | достаточно для контроля галлюцинаций                          |

* Embeddings - отдельный класс моделей, не входит в линейку Claude. В реальном проекте это будет внешний провайдер
* (Voyage, либо self-hosted модель). Поиск идёт по векторной БД (например pgvector или Qdrant)

## 4. Логика роутинга моделей (data-driven)

Простая, но рабочая схема: сначала дешёвый классификатор оценивает сложность,
дальше выбираем модель.

```
запрос
  |
  v
[Haiku 4.5: классификатор сложности и темы]
  |
  +-- простой типовой вопрос (есть прямой ответ в FAQ)
  |        -> Haiku 4.5 формирует короткий ответ
  |
  +-- средний вопрос (нужен синтез из 2-4 документов)
  |        -> Sonnet 4.6
  |
  +-- сложный / противоречивый / много источников
  |        -> Opus 4.8
  |
  +-- вне базы знаний / высокий риск / низкая уверенность
           -> эскалация на оператора (модель не отвечает сама)
```

### Сигналы для роутинга:

- метка сложности от классификатора
- число и качество найденных фрагментов (score из векторного поиска)
- порог уверенности (если максимальный score ниже порога - в базе скорее всего
  нет ответа, эскалируем)
- наличие противоречий между документами

### Data-driven часть:

Пороги и распределение по моделям не выставляем "на глаз", а подбираем на размеченной выборке реальных обращений
(held-out набор). Метрики - доля корректных ответов, доля галлюцинаций, стоимость на обращение, доля эскалаций.
Со временем логи накапливаются и пороги пересматриваются.

## 5. Защита от Dead Loop (зацикливания)

Агент в цикле Reason-Act-Observe может застрять. Защита многоуровневая:

- **счётчик итераций** - жёсткий лимит шагов на одно обращение (например 5-6). При
  достижении - принудительная эскалация
- **бюджет токенов на запрос** - если суммарный расход токенов превысил лимит,
  цикл останавливается
- **таймаут** - общее время обработки обращения ограничено
- **детектор повторов** - если агент два-три раза подряд выбирает одно и то же
  действие с теми же аргументами и не продвигается, считаем это тупиком
- **критерий прогресса** - на каждом витке должен появляться новый контекст или
  новое решение. Нет прогресса - выходим из цикла
- **fallback-ответ** - если уверенно ответить не вышло, агент не гадает, а отдаёт
  обращение человеку с уже собранным контекстом

## 6. Лимиты провайдера и где агент свалится

### Лимиты Anthropic (зависят от тарифного tier):

- ограничение запросов в минуту (RPM)
- ограничение токенов в минуту на вход и выход (ITPM/OTPM)
- ограничение токенов в сутки (TPD)
- размер контекстного окна: Opus 4.8 / Sonnet 4.6 - до 1M токенов,
  Haiku 4.5 - 200K
- максимум выходных токенов на ответ (для больших ответов нужен стриминг)

### Точки отказа и защита:

| Где падает               | Что делаем                                                                                 |
|--------------------------|--------------------------------------------------------------------------------------------|
| 429 (rate limit)         | повторы с экспоненциальной задержкой, очередь, при затяжке - откат на более дешёвую модель |
| 5xx / overloaded (529)   | ретраи с backoff, ретрай уже встроен в SDK                                                 |
| переполнение контекста   | обрезка истории, компакция, ограничение числа документов в контексте                       |
| таймаут сети             | повтор запроса, для длинных ответов - стриминг                                             |
| в базе знаний нет ответа | честный ответ "информации нет" + эскалация                                                 |
| противоречивые документы | пометка противоречия, эскалация на оператора                                               |
| недоступна векторная БД  | деградация - либо эскалация, либо ответ без RAG с явной пометкой                           |

Все внешние вызовы (LLM API, embeddings, векторная БД) обёрнуты в ретраи. 
Если по ходу обработки в БД создавались сущности (черновик тикета, лог), при сбое они откатываются, 
чтобы не оставалось мусора.

## 7. Системный промпт - что в нём лежит

Системный промпт собирается из стабильной и переменной частей:

- **инструкции агента** - роль, правила (отвечать только по контексту, цитировать
  источники, не выдумывать, формат ответа, когда эскалировать) - порядка 1000-1500 токенов
- **найденный контекст из базы знаний** - 3-5 релевантных фрагментов документов, порядка 1500-2500 токенов
- **немного few-shot примеров** хороших ответов и корректных эскалаций
- **история диалога** (если многоходовой) - переменная часть

Стабильную часть (инструкции + примеры) держим в начале и кешируем через prompt caching - это сильно экономит на 
повторных запросах. Переменное (контекст под конкретный запрос, история) идёт после точки кеширования.

Базу знаний целиком в промпт не кладём - она большая, поэтому и нужен RAG: в контекст попадают только релевантные куски.

## 8. Прикидка экономики

Цены Claude (за 1 млн токенов, вход / выход):

| Модель            | Вход  | Выход  |
|-------------------|-------|--------|
| Claude Haiku 4.5  | $1.00 | $5.00  |
| Claude Sonnet 4.6 | $3.00 | $15.00 |
| Claude Opus 4.8   | $5.00 | $25.00 |

Прикинем стоимость одного типового обращения (без учёта кеша, грубо):

1. Классификатор (Haiku): вход ~1000 ток, выход ~50 ток
    - 1000/1e6 * $1 + 50/1e6 * $5 = $0.001 + $0.00025 ~ $0.0013
2. Ответ на типовой вопрос (Sonnet): вход ~4000 ток (инструкции + 4 фрагмента),
   выход ~500 ток
    - 4000/1e6 * $3 + 500/1e6 * $15 = $0.012 + $0.0075 ~ $0.0195
3. Самопроверка (Sonnet): вход ~2000, выход ~100 ~ $0.0075

Итого порядка $0.028 за типовое обращение. Для сложного обращения с Opus 4.8
выйдет дороже - примерно $0.05-0.08.

С prompt caching на стабильной части системного промпта вход по кешу стоит около
0.1x от обычной цены, и реальная стоимость падает в несколько раз (особенно при
большом потоке однотипных запросов). Запись в кэш при этом дороже обычного входа
примерно в 1.25 раза (для TTL 5 минут), поэтому кэш окупается начиная со второго
обращения с тем же префиксом.

Грубая оценка при 10 000 обращений в месяц и среднем чеке $0.03:

- без кеша: ~$300 / мес
- с кешированием стабильной части: ориентировочно $120-180 / мес

В рублях (по курсу примерно 90) это около 11-16 тыс. руб. в месяц - заметно дешевле, 
чем ручная обработка тех же объёмов первой линией.

### Главные рычаги экономии:

- роутинг (дешевая модель закрывает большую часть простых запросов)
- prompt caching стабильного системного промпта
- ограничение числа документов в контексте
- лимит итераций цикла

## 9. Скорость первого ответа (TTFT не более 500 мсек)

За 500 мсек полный конвейер (классификация, retrieval, генерация) отработать не успеет -
один только векторный поиск с реранкингом обычно занимает больше. Поэтому требование
закрываем комбинацией приёмов:

- **стриминг** - ответ отдаётся пользователю токен за токеном, ждать полной генерации не нужно
- **мгновенная первая фраза** - сразу после получения обращения пользователю уходит короткое
  подтверждение ("Секунду, уточняю по базе знаний"). Берётся из набора готовых шаблонов или
  генерируется Haiku 4.5 по полностью закешированному промпту - оба варианта укладываются в лимит
- **параллельный запуск конвейера** - пока пользователь читает первую фразу, уже идут
  классификация и retrieval, основной ответ дописывается следом в тот же стрим
- **prompt caching** - стабильная часть системного промпта лежит в кэше, это заметно
  сокращает время до первого токена (TTFT)
- **прогрев кэша** - у кэша TTL 5 минут, при редком трафике держим его тёплым периодическим
  прогревом, иначе первая фраза после долгой паузы рискует не уложиться в лимит
- **мониторинг TTFT** - следим за p95 времени первого токена, при деградации - алерт

Отдельный случай - 429/5xx от провайдера: ретраи с backoff удлиняют основной ответ, но первая
фраза из шаблона от провайдера не зависит и отдаётся всегда.

## 10. Итоговая архитектура (кратко)

```
Пользователь
    |
    v
[Оркестратор агента] -- цикл Reason/Act/Observe, лимиты, ретраи
    |          |              |                 |
    v          v              v                 v
[Классиф.] [Retrieval]   [Генерация]      [Эскалация]
 Haiku 4.5   |            Sonnet/Opus       -> оператор
             v
   [Embeddings (Voyage)] + [Векторная БД] <- [База знаний]
    |
    v
[Логирование, метрики, накопление выборки для data-driven роутинга]
```

### Принцип:

Дешёвая модель решает что делать и закрывает простое, средняя отвечает на типовое, продвинутая включается только на
сложном, а спорное и рискованное уходит человеку. Все ответы опираются на найденный контекст и снабжены ссылками
на источники.
