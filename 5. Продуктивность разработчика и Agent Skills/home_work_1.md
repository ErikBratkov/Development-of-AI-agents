# create-agent-tool

Запись через слэш (`/create-agent-tool`) - это нотация конкретного клиента.
В Anthropic API skill не вызывается командой, а подключается моделью по описанию
из frontmatter, когда задача подходит под триггер (discovery)

## Назначение

SOP описывает повторяемый процесс создания нового инструмента (tool) для агента -
от формулировки потребности до готовой к подключению функции с описанием схемы,
обработкой ошибок и тестами. Цель - чтобы любой инструмент в проекте делался по
одному стандарту и был предсказуем для модели и для разработчика.

## 1. Trigger - когда применять

Запускаем процедуру, когда выполнено хотя бы одно из условий:

- агенту в диалоге нехватает данных или возможности выполнить действие
  (получить курс валют, сходить в базу, отправить письмо, посчитать что-то детерминированно)
- одна и та же ручная операция повторяется в нескольких сценариях и ее стоит вынести в код
- модель пытается "придумать" результат вместо реального вызова (галлюцинирует данные)
- нужно детерминированное и проверяемое поведение там, где LLM ненадежна
  (математика, доступ к актуальным данным, побочные эффекты)

Не применяем, если задачу можно решить промптом или это разовое действие без повторов

## 2. Inputs - что нужно знать до старта

Перед началом собираем:

- бизнес-цель инструмента в одном предложении (что и зачем он делает)
- сигнатура: входные аргументы с типами и единицами измерения, формат результата
- источник данных или внешний сервис (URL API, таблица БД, библиотека) и способ аутентификации
- ограничения: лимиты запросов, таймауты, стоимость вызова, требования к идемпотентности
- сценарии отказа внешней системы и желаемое поведение при них (повтор, откат, заглушка)
- кто потребитель инструмента - конкретный агент, и как он будет его вызывать

## 3. Procedure - шаги

1. Сформулировать назначение и однозначный контракт: имя, аргументы, типы, что
   возвращает. Имя глагольное и говорящее (get_weather, create_invoice)
2. Спроектировать JSON-схему параметров: обязательные и опциональные поля, описания
   каждого аргумента на понятном языке - это то, по чему модель решает когда звать tool
3. Реализовать чистую функцию с аннотацией типов и docstring. Бизнес-логику отделить
   от слоя вызова модели, побочные эффекты держать в одном месте
4. Добавить обработку отказов внешнего API: таймауты, повторы с backoff, откат уже
   созданных сущностей в БД, понятное сообщение об ошибке вместо исключения наружу
5. Валидировать вход и выход: проверить аргументы до вызова API, привести результат
   к описанной схеме, не отдавать модели сырой неструктурированный ответ
6. Зарегистрировать инструмент в наборе tools агента и описать его в системном
   промпте или реестре, чтобы модель знала о его существовании
7. Написать тесты (unit с моками внешнего сервиса, при необходимости integration)
   и проверить вызов на 2-3 реальных запросах от лица агента

## 4. Verification - как доказать готовность

- [ ] У инструмента есть имя, описание и JSON-схема параметров
- [ ] Функция типизирована, имеет docstring и не мутирует входные аргументы
- [ ] Покрыт сценарий полного и частичного отказа внешнего API (повтор/откат)
- [ ] Есть unit-тесты с моками, покрытие не ниже 80 процентов
- [ ] Агент реально вызывает tool в нужный момент, а не выдумывает ответ
- [ ] Результат возвращается в заявленном формате и проходит валидацию
- [ ] Инструмент зарегистрирован в наборе tools и виден модели

## 5. Anti-patterns - что нельзя делать

- описывать аргументы расплывчато ("data", "params") - модель не поймет когда звать tool
- складывать в один инструмент несколько несвязанных действий
- молча проглатывать ошибки внешнего API или возвращать модели стектрейс
- оставлять побочные эффекты без отката при частичном сбое
- хардкодить секреты и ключи прямо в коде инструмента
- доверять LLM то, что должно быть детерминированным (расчеты, форматирование, доступ к данным)
- выкатывать инструмент без тестов и без проверки реального вызова агентом

## 6. Граница между skill и tool

Skill - это повторяемая процедура (SOP), которая живет на уровне рассуждений модели:
последовательность шагов, критерии выбора, проверки результата. Skill отвечает на
вопрос "как действовать" и остается текстом-инструкцией.

Tool - это конкретное детерминированное действие, вынесенное в код. Tool отвечает на
вопрос "чем именно выполнить шаг" и дает гарантированный, проверяемый результат.

Что остается процедурой (skill):
- решение, какой инструмент и когда вызвать
- интерпретация результата и формулировка ответа пользователю
- ветвление сценария, уточняющие вопросы, синтез из нескольких источников

Что надо автоматизировать кодом (tool):
- вызовы внешних API и запросы в БД
- математика и любые точные вычисления
- работа с файлами, побочные эффекты, операции с откатом
- все, где недопустима галлюцинация и нужен стабильный формат

Простое правило: если шаг требует точности, доступа к данным или побочного эффекта -
это tool. Если шаг про выбор, оценку и порядок действий - это skill.

## Ответ на вопрос для самопроверки

Skill - это не просто сохраненный промпт, потому что он задает не один текст, а
повторяемый workflow: явный триггер (когда применять), нужные входные данные,
последовательность шагов и - главное - блок verification с критериями готовности.
Сохраненный промпт выдает результат "как получится" и его качество ничем не
гарантировано, а skill описывает процесс и сразу проверяет, что результат
действительно достигнут, и отделяет шаги для LLM-логики от шагов, которые надежнее
отдать в код (tool). Поэтому skill воспроизводим и его можно улучшать как процесс,
а не переписывать промпт каждый раз заново.

## Адаптация SOP под Anthropic function calling (Python)

Ниже тот же процесс, но привязанный к конкретике Anthropic Messages API и SDK
`anthropic` для Python. Модель по умолчанию - `claude-opus-4-8`.

### Как устроен tool use в Anthropic

- инструмент описывается тремя полями: `name`, `description`, `input_schema`
  (JSON Schema аргументов). Описание - это то, по чему модель решает когда звать tool
- модель не выполняет инструмент сама. Она возвращает в ответе блок `tool_use`
  с полями `id`, `name`, `input`, и `response.stop_reason` становится `"tool_use"`
- наш код выполняет функцию и отправляет назад блок `tool_result` с тем же
  `tool_use_id`. При ошибке ставим `is_error: True` и кладем текст ошибки
- цикл повторяется пока `stop_reason` не станет `"end_turn"` - это и есть
  агентный цикл (agentic loop)

### Что меняется в шагах процедуры

1. Контракт инструмента превращается в `input_schema` (JSON Schema): обязательные
   поля в `required`, для перечислимых значений - `enum`, `additionalProperties: false`
2. Имя и `description` пишем прескриптивно: не только что делает tool, но и когда
   его вызывать - современные модели зовут инструменты осторожно
3. Реализуем чистую Python-функцию-обработчик с аннотацией типов и docstring
4. Отказ внешнего API не пробрасываем исключением наружу: ловим, возвращаем
   `tool_result` с `is_error: True` и понятным текстом, чтобы модель могла
   среагировать. SDK сам делает повторы по 429 и 5xx (параметр `max_retries`)
5. Парсим `block.input` как готовый dict (SDK уже распарсил JSON), но валидируем
   аргументы до вызова внешнего сервиса
6. Регистрируем инструмент в списке `tools` запроса к `client.messages.create`
7. Тесты: unit с моком внешнего сервиса плюс проверка реального вызова от лица модели

Для рассуждающих моделей семейства opus-4-8 в запрос имеет смысл добавить
`thinking={"type": "adaptive"}` - модель сама решает сколько думать перед
вызовом инструмента, отдельный бюджет токенов задавать не нужно

### Verification применительно к API

- [ ] в ответе появляется блок `tool_use` с ожидаемым `name` и валидным `input`
- [ ] на каждый `tool_use` уходит `tool_result` с совпадающим `tool_use_id`
- [ ] сценарий отказа возвращает `is_error: True`, а не роняет процесс
- [ ] цикл завершается по `stop_reason == "end_turn"` и не зацикливается
- [ ] финальный ответ модели опирается на результат инструмента, а не выдуман

### Граница skill/tool в терминах API

Tool - это запись в массиве `tools` плюс Python-обработчик: детерминированное
действие с гарантированным форматом (вызов API, расчет, доступ к данным). Skill -
это системный промпт и логика цикла: когда какой инструмент позвать, как
истолковать `tool_result` и что ответить пользователю. Все что должно быть точным
и иметь побочный эффект - в обработчик tool, все что про выбор и интерпретацию -
в промпт и оркестрацию.

## Шаблон кода инструмента

```python
import json
from typing import Any

import anthropic

# имя модели по умолчанию
MODEL_NAME = "claude-opus-4-8"

# системный промпт - это зона skill: описываем когда звать инструмент и как
# трактовать результат. Решение позвать tool и формулировку ответа отдаем модели
SYSTEM_PROMPT = (
    "Ты ассистент по погоде. Когда пользователь спрашивает про погоду в "
    "городе - вызывай инструмент get_weather и не выдумывай данные. Получив "
    "результат инструмента, сформулируй короткий понятный ответ на русском"
)

# описание инструмента для модели. От качества description напрямую зависит
# то, насколько верно модель будет решать когда звать tool
WEATHER_TOOL: dict[str, Any] = {
    "name": "get_weather",
    "description": (
        "Вернуть текущую погоду для города. "
        "Вызывай, когда пользователь спрашивает про погоду, температуру "
        "или условия в конкретном населенном пункте"
    ),
    "input_schema": {
        "type": "object",
        "properties": {
            "city": {
                "type": "string",
                "description": "Название города, например Москва",
            },
            "unit": {
                "type": "string",
                "enum": ["celsius", "fahrenheit"],
                "description": "Единица измерения температуры",
            },
        },
        "required": ["city"],
        "additionalProperties": False,
    },
}


def fetch_weather(city: str, unit: str = "celsius") -> dict[str, Any]:
    """Сходить во внешний сервис погоды и вернуть результат словарем

    Чистая по отношению к аргументам функция - входные данные не меняет.
    Здесь должен быть реальный вызов API. Для примера возвращаем заглушку
    """
    # тут был бы requests.get(...) или клиент внешнего сервиса
    temperature = 21 if unit == "celsius" else 70
    return {"city": city, "unit": unit, "temperature": temperature}


def run_weather_tool(tool_input: dict[str, Any]) -> dict[str, Any]:
    """Выполнить инструмент get_weather и вернуть блок tool_result

    Валидирует вход, обрабатывает отказ внешнего сервиса и всегда возвращает
    корректный по структуре результат - либо данные, либо is_error
    """
    city = tool_input.get("city")
    unit = tool_input.get("unit", "celsius")

    # валидация до обращения к внешнему сервису
    if not city:
        return {
            "content": "Ошибка: не передан обязательный аргумент city",
            "is_error": True,
        }

    try:
        data = fetch_weather(city, unit)
    except Exception as err:
        # отказ внешнего API не роняем наружу, отдаем модели текст ошибки
        return {
            "content": f"Ошибка обращения к сервису погоды: {err}",
            "is_error": True,
        }

    return {"content": json.dumps(data, ensure_ascii=False), "is_error": False}


# сюда складываем все доступные инструменты по имени
TOOL_HANDLERS = {"get_weather": run_weather_tool}


def ask_with_tools(user_text: str, max_steps: int = 5) -> str:
    """Прогнать запрос пользователя через агентный цикл с инструментами

    Крутим цикл пока модель не закончит. На каждый блок tool_use вызываем
    нужный обработчик и возвращаем tool_result обратно в диалог
    """
    client = anthropic.Anthropic()
    messages: list[dict[str, Any]] = [{"role": "user", "content": user_text}]
    last_text = ""

    for _ in range(max_steps):
        response = client.messages.create(
            model=MODEL_NAME,
            max_tokens=1024,
            system=SYSTEM_PROMPT,
            thinking={"type": "adaptive"},
            tools=[WEATHER_TOOL],
            messages=messages,
        )

        # запоминаем текст ассистента - пригодится если упремся в лимит шагов
        step_text = "".join(b.text for b in response.content if b.type == "text")
        if step_text:
            last_text = step_text

        # модель закончила - отдаем собранный текст
        if response.stop_reason == "end_turn":
            # на случай ответа без текстовых блоков отдаем понятный фолбэк
            return step_text or "Модель завершила ход без текстового ответа"

        if response.stop_reason != "tool_use":
            # необработанная причина остановки (например max_tokens) - не зацикливаемся,
            # но отдаем уже накопленный текст если он есть
            return step_text or f"Остановка по причине {response.stop_reason}"

        # сохраняем ответ ассистента целиком, включая блоки tool_use
        messages.append({"role": "assistant", "content": response.content})

        # выполняем каждый запрошенный инструмент
        tool_results = []
        for block in response.content:
            if block.type != "tool_use":
                continue
            handler = TOOL_HANDLERS.get(block.name)
            if handler is None:
                result = {
                    "content": f"Неизвестный инструмент: {block.name}",
                    "is_error": True,
                }
            else:
                # block.input уже распарсен SDK в dict, json.loads не нужен
                result = handler(block.input)
            tool_result: dict[str, Any] = {
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": result["content"],
            }
            # is_error по конвенции ставим только когда инструмент реально упал
            if result["is_error"]:
                tool_result["is_error"] = True
            tool_results.append(tool_result)

        # отдаем результаты инструментов назад одним сообщением
        messages.append({"role": "user", "content": tool_results})

    # шаги исчерпаны - возвращаем последний полезный текст, а не глотаем его
    return last_text or "Не удалось получить финальный ответ в отведенное число шагов"


if __name__ == "__main__":
    print(ask_with_tools("Какая погода в Москве?"))
```

Замечание про границу: функции `fetch_weather` и `run_weather_tool` - это tool
(детерминированный код с обработкой отказа), а решение позвать `get_weather` и
формулировка итогового ответа - работа модели внутри `ask_with_tools`, то есть
зона skill. Если инструментов много, имеет смысл вынести описания в реестр и
рассмотреть `client.beta.messages.tool_runner` с декоратором `@beta_tool`, который
сам ведет агентный цикл.
