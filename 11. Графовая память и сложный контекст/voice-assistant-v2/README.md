# voice-assistant-v2

Учебный демо-сервис голосового ассистента (ДЗ к урокам 9 и 11).
Монолит на FastAPI: браузер шлет голос по WebSocket, сервер распознает
его локально (faster-whisper), отправляет текст в LLM через OpenRouter
и стримит обратно токены ответа и озвучку (Piper). Контекст диалога
живет в RAM, при разрастании сжимается в rolling summary.

К уроку 11 добавлена база знаний в Neo4j: лексический граф (документы,
нарезанные на фрагменты с эмбеддингами и векторным индексом) плюс
доменный граф (люди, компании, технологии, проекты и связи между
ними). Данные генерирует пакет из папки `data-generator`, наполнение
базы делает скрипт `seed_knowledge.py`. Модель отвечает на вопросы
по базе через гибридный поиск - инструмент search_knowledge,
устроенный по аналогии с get_weather.

Подробности архитектуры - в ../ARCHITECTURE.md.

## Соответствие критериям ДЗ

Урок 9:

- передача контекста - источником знаний выступает сама LLM, контекст
  диалога (system prompt, summary, последние реплики) собирается в
  `app/memory.py` и передается в модель на каждом ходе
- streaming ответа - токены LLM и аудио TTS стримятся в браузер по
  одному WebSocket по мере генерации
- воспроизводимость - кроме голоса есть текстовый ввод, он работает
  без микрофона и без аудио-зависимостей
- function calling - у модели есть инструмент get_weather
  (`app/weather.py`), на вопрос про погоду в городе она сама зовет
  Open-Meteo API (геокодинг названия плюс текущая погода) и отвечает
  по полученным данным, цикл вызова инструментов живет в
  `app/llm_client.py`

Урок 11:

- работает поиск по данным - векторный поиск по фрагментам документов
  через нативный векторный индекс Neo4j, вопрос эмбеддится локальной
  моделью (`app/embeddings.py`)
- реализованы связи между объектами - доменный граф с отношениями
  WORKS_AT, KNOWS, WORKS_ON, OWNED_BY, USES, DEPENDS_ON генерируется
  пакетом `data-generator` и грузится скриптом `seed_knowledge.py`
- гибридный поиск - `app/knowledge.py` сначала находит фрагменты по
  смыслу, затем добирает факты обходом графа от упомянутых сущностей
- показан результат ответа - пример запроса и ответа в разделе
  "База знаний в Neo4j" ниже

## Установка

Нужен Python 3.13

Быстрый способ - готовые скрипты, они создают .venv рядом с собой,
ставят зависимости (полный вариант включает и группу rag для базы
знаний), копируют .env.example в .env и при полном варианте
скачивают голос для Piper:

```bash
# Linux / macOS
./install_linux.sh          # полный вариант с STT и TTS
./install_linux.sh min      # минимальный вариант, только текст
```

```powershell
# Windows 11
install_windows.cmd         # полный вариант с STT и TTS
install_windows.cmd min     # минимальный вариант, только текст
```

После установки впишите свой OPENROUTER_API_KEY в .env.

Дальше в этом разделе те же шаги описаны для ручной установки.

Создание и активация виртуального окружения.

### Linux / macOS:

```bash
cd voice-assistant-v2
python3 -m venv .venv
source .venv/bin/activate
```

### Windows 11 (PowerShell или cmd):

Важно:
в пути проекта не должно быть кириллических символов

```powershell
cd voice-assistant-v2
py -3.13 -m venv .venv
.venv\Scripts\activate
```

Если PowerShell ругается на политику выполнения скриптов, разрешите
локальные скрипты для текущего пользователя:

```powershell
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
```

Установка зависимостей (одинаково для всех ОС):

```bash
# минимальный вариант - только текстовый режим
pip install -e .

# полный вариант - с локальным STT и TTS
pip install -e ".[audio]"

# база знаний в Neo4j и локальные эмбеддинги (гибридный поиск),
# без этой группы инструмент search_knowledge вежливо откажет
pip install -e ".[rag]"

# группа dev добавляет pytest для unit тестов, скрипты установки
# ставят ее сами, при ручной установке комбинируйте с нужным вариантом
pip install -e ".[dev]"
pip install -e ".[audio,rag,dev]"
```

Конфигурация:

```bash
# Linux / macOS / PowerShell
cp .env.example .env
# в cmd вместо cp используйте: copy .env.example .env
# впишите свой OPENROUTER_API_KEY
```

Голос для Piper (только для полного варианта):

```bash
mkdir models
python -m piper.download_voices ru_RU-irina-medium --data-dir models
# путь к .onnx файлу укажите в TTS_VOICE в .env
```

Модель Whisper скачается сама при первом распознавании, поэтому
первый голосовой ход может занять заметное время.

## База знаний в Neo4j

Нужны docker и зависимости из группы rag (см. установку выше).

Скрипты запуска (run_linux.sh, run_windows.cmd) поднимают контейнер
и наполняют базу сами. Ручной вариант:

```bash
docker compose up -d --wait
python seed_knowledge.py
```

Сам набор данных делает пакет генератора из папки `data-generator`
(подробности в его README), скрипт лишь добавляет эмбеддинги и
векторный индекс. Генерация идет с фиксированным seed: 50
сотрудников, 6 компаний, 12 технологий, 15 проектов, связи между
ними и документы (профиль сотрудника, справка о проекте), нарезанные
на фрагменты. Тексты фрагментов эмбеддятся локальной моделью
(intfloat/multilingual-e5-small, скачается при первом запуске),
поверх создается векторный индекс. Скрипт можно запускать повторно,
он всегда начинает с очистки графа.

Хитрость набора данных: кто где работает и на каких проектах, в
текстах документов не написано - эти факты живут только в связях
графа. Документы же знают про должности, опыт и навыки. Поэтому на
вопросы вида "кто знает X и где он работает" чисто векторный поиск
ответить не может, а гибридный - может.

Веб-интерфейс Neo4j доступен на http://localhost:7474 (логин neo4j,
пароль graphdemo42 из docker-compose.yml).

### Пример запроса и ответа

Вопрос ассистенту (текстом или голосом):

    Кто из сотрудников знает Neo4j и в каких компаниях они работают?

Модель сама вызывает инструмент search_knowledge. Тот эмбеддит
вопрос, векторным поиском находит фрагменты профилей сотрудников с
упоминанием Neo4j, затем обходом графа добирает связи WORKS_AT и
WORKS_ON найденых людей. В контекст модели уходит примерно такой
результат инструмента:

    Факты из графа знаний:
    - Shannon Ray -[WORKS_AT]-> Globex
    - Shannon Ray -[KNOWS]-> Neo4j
    - Daniel Adams -[WORKS_AT]-> Acme
    - Daniel Adams -[KNOWS]-> Neo4j
    ...

    Выдержки из базы знаний:
    - Shannon Ray profile: Shannon Ray has hands on experience
      with Neo4j, FastAPI, Kubernetes.
    ...

И ассистент отвечает в духе:

    Neo4j знают Shannon Ray (работает в Globex) и Daniel Adams
    (работает в Acme).

Конкретные имена зависят от сгенерированного набора, но при
неизменном seed они воспроизводятся от запуска к запуску. Ключевой
момент: принадлежность людей к компаниям модель узнает только из
связей графа, в текстах заметок ее нет.

## Запуск

### Linux / macOS:

```bash
./run_linux.sh
```

Скрипт активирует .venv, поднимает контейнер Neo4j (docker compose
up -d --wait, ключ ждет готовности базы по healthcheck), наполняет
базу знаний и запускает сервер, открыв http://localhost:8000 в
браузере. Если docker недоступен, сервис стартует без базы знаний -
инструмент search_knowledge вежливо откажет. Ручной вариант -
активировать окружение и выполнить `python main.py`.

Контейнер Neo4j после выхода из сервера продолжает работать,
остановить его можно скриптом:

```bash
./stop_linux.sh
```

Данные базы живут в volume и остановку переживают.

Замечание про системный прокси: рабочие столы Linux нередко задают
all_proxy со схемой socks://, которую httpx не понимает. Сервис и
seed-скрипт на старте сами переписывают ее в socks5://, поддержка
socks ставится вместе с зависимостями (httpx[socks]).

Замечание про микрофон: getUserMedia работает только в secure
context, то есть на https или на localhost. Если открываете сервис
по адресу вида http://192.168.x.x, голос будет недоступен и
останется текстовый режим.

### Windows 11 (PowerShell или cmd):

Запустите run_windows.cmd, поведение то же, что у run_linux.sh.
Остановка контейнера Neo4j после работы - stop_windows.cmd

## Сценарий для демонстрации

1. Откройте страницу, дождитесь статуса "готов"
2. Зажмите кнопку, скажите "Привет, меня зовут Эрик", отпустите
3. Дождитесь озвученного ответа
4. Спросите голосом или текстом "Как меня зовут?" - ассистент помнит
   контекст и ответит правильно
5. Для перебивания зажмите кнопку прямо во время ответа - генерация
   и озвучка оборвутся, начнется новый ход
6. Если база знаний наполнена, спросите "Кто из сотрудников знает
   Neo4j и в каких компаниях они работают?" - модель вызовет
   search_knowledge и соберет ответ из заметок и связей графа

## Протокол WebSocket

Одно соединение `/ws`, кадры двух типов. Бинарные: вверх - PCM16
16 kHz с микрофона, вниз - PCM16 озвучки. Текстовые - JSON с полем
`type`:

- от клиента: `start`, `stop`, `barge_in`, `text_input`
- от сервера: `ready`, `kb_loading`, `kb_ready`, `final_transcript`,
  `llm_token`, `tts_start`, `tts_end`, `turn_done`, `error`

`text_input` - текстовый fallback, в остальном ход обрабатывается
так же, как голосовой.

`kb_loading` приходит сразу после `ready`, если на сервере еще
грузится модель поиска по базе знаний - клиент на это время
блокирует ввод и показывает статус загрузки. `kb_ready` снимает
блокировку.

## Структура

```text
main.py                   # FastAPI, /ws, отдача статики, lifespan
docker-compose.yml        # контейнер Neo4j (порты 7474 и 7687)
seed_knowledge.py         # эмбеддинги и загрузка датасета в Neo4j
data-generator/           # подкинутый генератор графовых данных,
                          # seed_knowledge.py берет датасет из него
app/
├── config.py             # pydantic-settings, чтение .env
├── ws_gateway.py         # websocket, буфер аудио, маршрутизация
├── dialogue_manager.py   # контекст, оркестрация хода, barge-in
├── memory.py             # история, rolling summary, бюджет токенов
├── llm_client.py         # OpenRouter streaming, ретраи, инструменты
├── tools.py              # реестр инструментов для function calling
├── weather.py            # инструмент get_weather, Open-Meteo API
├── knowledge.py          # инструмент search_knowledge, гибридный поиск
├── embeddings.py         # локальные эмбеддинги, sentence-transformers
├── stt.py                # faster-whisper
├── tts.py                # piper
└── static/               # фронтенд без сборщика
tests/                    # unit тесты сервиса
```
